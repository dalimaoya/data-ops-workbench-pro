## 二、逐项测试结果

### 1. 模板底部空白区是否可正常填写新数据
**结果：✅ 通过**

代码审查：`export_template()` 函数中，`RESERVED_BLANK_ROWS = 50`，在数据行之后生成 50 行空白行。空白区域中：
- 主键列设为 `unlocked`（`cell.protection = unlocked_cell`），允许填写新主键
- 可编辑字段设为 `unlocked`
- 不可编辑字段设为 `locked`
- 空白行背景色为浅黄色 `#FFFFF0`（`blank_zone_fill`），与数据行视觉区分

结论：空白区域设计合理，主键和可编辑字段均可填写。

---

### 2. 新增行填写后导入，校验是否正确识别为新增行
**结果：✅ 通过**

代码审查：`import_template()` 第 7 步"Generate diff"逻辑：
1. 从数据库查询所有现有行，构建 `db_pk_map`
2. 遍历导入的每行数据，检查 `pk_key not in db_all_pk_set`
3. 若 PK 不在数据库中且 `tc.allow_insert_rows` 为真，则标记为 `new_rows`，`change_type = "insert"`
4. diff_rows 中记录所有非空字段，`status = "new"`，`change_type = "insert"`

结论：新增行识别逻辑正确，通过主键与数据库比对实现。

---

### 3. 新增行主键不在数据库中 → 识别为 INSERT
**结果：✅ 通过**

代码位置：`import_template()` 约第 500 行：
```python
if pk_key not in db_all_pk_set:
    if tc.allow_insert_rows:
        new_rows.append({...change_type: "insert"})
```
当主键不在 DB 中且允许新增时，正确识别为 INSERT。

---

### 4. 新增行主键已在数据库中 → 应识别为 UPDATE（非新增）
**结果：✅ 通过**

代码逻辑：当 `pk_key in db_all_pk_set` 时，进入"已有行"分支，对比差异字段，`change_type = "update"`。不会错误地归入新增行。

### 6. 回写执行 INSERT 后，数据确实写入业务表
**结果：✅ 通过**

代码审查：`writeback()` 函数"Step 3: Execute INSERTs"：
1. 从 `new_rows_data` 获取新增行列表
2. 构建 INSERT SQL：`INSERT INTO {qt} ({insert_cols_sql}) VALUES ({placeholders})`
3. 使用 `_exec()` 执行，兼容多数据库（MySQL、PostgreSQL、Oracle、SQL Server、DM、SQLite）
4. 执行后 `conn.commit()` 提交事务
5. 成功时记录 `FieldChangeLog`，`change_type = "insert"`

结论：INSERT 回写逻辑完整，包含事务提交和变更日志记录。

### 8. 混合场景：同一批次含更新行 + 新增行
**结果：✅ 通过**

代码审查：`writeback()` 分两步执行：
1. Step 2 处理更新行（`update_rows` = 不在 `new_pk_set` 中的行）
2. Step 3 处理新增行（`new_rows_data`）
3. 分别计数 `update_count` 和 `insert_count`
4. 最终 `success_count = update_count + insert_count`

两类操作在同一事务中执行，最后统一 `conn.commit()`。失败不互相影响（逐行 try/except）。

结论：混合场景处理正确。

### 9. 边界：新增行缺少必填字段 → 校验报错
**结果：✅ 通过**

代码审查：`import_template()` 第 6 步"Read rows & validate"：
```python
if fc.is_required and (str_val is None or str_val == ""):
    row_errors.append({...type: "required"...})
```

此校验对所有行生效（包括新增行），缺少必填字段会产生 `required` 类型错误，该行会被标记为失败，不会进入回写。

结论：必填字段校验覆盖新增行。

### 11. 边界：新增行超过模板预留空白行数（当前 50 行）时的表现
**结果：⚠️ 缺陷 — P2（中等）**

代码审查：
- 导出时 `RESERVED_BLANK_ROWS = 50`，空白行范围固定
- 导入时解析逻辑：`for row_idx in range(2, data_ws.max_row + 1)`，遍历到 Excel 最大行
- 模板启用了 Sheet Protection（`insertRows=False`），**禁止用户插入新行**

**问题**：用户在模板中填满 50 行空白行后，无法再添加更多新增行（因为 Sheet Protection 阻止插入行）。但如果用户绕过保护（如移除保护密码），导入代码本身不限制新增行数，因此：
- 正常情况：受模板保护限制，最多 50 行新增 → **功能正常但缺少明确提示**
- 异常情况：若用户破解保护插入更多行，导入仍可正常处理 → 无崩溃风险

**缺陷描述**：模板中预留空白行数（50 行）为硬编码值，无配置入口。当用户需要新增超过 50 行时：
1. 没有任何前端提示告知用户上限
2. 没有后端配置项调整此值
3. 导出模板时不会在 Excel 中提示"最多可新增 50 行"

**严重级别：P2（中等）**
**建议处理方式**：
1. 在模板空白区首行添加注释说明最大新增行数
2. 将 `RESERVED_BLANK_ROWS` 改为可配置项（TableConfig 级别）
3. 考虑在导入时增加提示：当新增行接近上限时给出警告

### 12. 权限：只读用户上传含新增行的模板 → 应被拒绝
**结果：⚠️ 缺陷 — P1（重要）**

代码审查：

**导入接口权限**：
```python
@router.post("/{table_config_id}/import")
async def import_template(..., user: UserAccount = Depends(require_role("admin", "operator"))):
```
`require_role("admin", "operator")` 限制了只有 admin 和 operator 角色可上传导入 → **导入本身已阻止只读用户** ✅

**回写接口权限**：
```python
@router.post("/import-tasks/{task_id}/writeback")
def writeback(..., user: UserAccount = Depends(require_role("admin", "operator"))):
```
同样限制 → ✅

**导出接口权限**：
```python
@router.post("/{table_config_id}/export")
def export_template(..., user: UserAccount = Depends(require_role("admin", "operator"))):
```
只读用户无法导出含空白行的模板 → ✅

**但存在问题**：
1. `list_maintenance_tables()` 返回的 `allow_insert_rows` 对只读用户返回 `0`（正确），viewer 看不到新增入口
2. **前端 DiffPreview.tsx 中**，`canWriteback` 判断为 `user?.role === 'admin' || user?.role === 'operator'`，viewer 无法点击回写按钮 → ✅

**发现的缺陷**：`allow_insert_rows` 字段在 `list_maintenance_tables` 返回时对 viewer 返回 0，但数据浏览接口 `browse_table_data` 返回的 `allow_delete_rows` 做了权限过滤，**没有同时返回 `allow_insert_rows` 的权限过滤结果**。

查看代码：
```python
# list_maintenance_tables 中:
"allow_insert_rows": row.allow_insert_rows if user.role in ("admin", "operator") else 0,
"allow_delete_rows": row.allow_delete_rows if user.role in ("admin", "operator") else 0,

# browse_table_data 中:
"allow_delete_rows": tc.allow_delete_rows if user.role in ("admin", "operator") else 0,
# ← 缺少 allow_insert_rows 的返回
```

**缺陷描述**：`browse_table_data` 接口返回了 `allow_delete_rows`（带权限过滤），但**没有返回 `allow_insert_rows`**。前端 `DataBrowse` 组件读取此接口来决定是否显示"新增行"按钮。虽然后端 API 层面阻止了只读用户的实际操作，但前端可能因缺少此字段而出现显示不一致。

**严重级别：P1（重要）** — 功能缺失，`browse_table_data` 缺少 `allow_insert_rows` 返回值。

**建议处理方式**：在 `browse_table_data` 返回值中增加：
```python
"allow_insert_rows": tc.allow_insert_rows if user.role in ("admin", "operator") else 0,
```
