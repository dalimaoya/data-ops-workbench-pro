## 1. 背景与现状

### 1.1 现有功能

**后端（plugin-sql-console）：**
- `POST /api/sql-console/execute` — 执行 SELECT 查询，返回列名 + 行数据，最多 1000 行
- `POST /api/sql-console/export` — 查询结果导出 Excel，最多 50000 行
- 安全限制：仅允许 SELECT / EXPLAIN / SHOW / DESCRIBE，阻止 INSERT/UPDATE/DELETE/DROP 等写操作
- 多语句禁止，SQL 注入关键字检测

**前端（SqlConsolePage.tsx）：**
- 数据源选择器
- 纯 `<textarea>` SQL 输入框（深色背景、等宽字体）
- Ctrl+Enter 执行快捷键
- 结果表格（Ant Design Table，分页 50 行）
- 导出 Excel 按钮
- 只读提示 Alert

**已有的 AI 自然语言查询（ai_nl_query.py）：**
- 面向**纳管表**的自然语言查询（需要 table_id，依赖 TableConfig/FieldConfig）
- 规则引擎：日期范围、枚举匹配、数值比较、否定语义
- LLM 引擎：发送字段上下文给大模型，解析为结构化 filter
- 输出结构化 filter 条件 + SQL 预览，非任意 SQL 生成

### 1.2 差距分析

| 维度 | 当前状态 | 期望状态 |
|---|---|---|
| SQL 编辑体验 | 纯 textarea，无语法高亮 | 语法高亮 + 基础自动补全 |
| 查询门槛 | 必须手写 SQL | 可视化选表选字段 + AI 辅助生成 |
| 查询范围 | 任意数据源任意表 | 保持（vs NL查询仅限纳管表） |
| 结果处理 | 表格展示 + Excel 导出 | 增加 CSV 导出 + 列统计 |
| 历史记录 | 无 | 保存最近查询，可快速复用 |
| AI 能力 | 无（NL查询是独立模块） | 集成 AI 辅助生成 SQL |

## 2. 设计原则

| 原则 | 说明 |
|---|---|
| 查询专用 | SQL 操作台只做 SELECT 查询，不开放写操作（DDL/DML 由库表管理和数据回写模块负责） |
| 三种入口并存 | 手写 SQL、可视化构建、AI 生成——三条路径生成 SQL，最终都通过同一个执行器运行 |
| SQL 透明 | 可视化和 AI 生成的 SQL 都先展示给用户确认，再执行——培养 SQL 认知 |
| 轻量不膨胀 | 不做成 DataGrip/DBeaver，够用即可——语法高亮用轻量方案，不引入重量级编辑器 |
| 复用已有 AI 基础设施 | 共用 AIEngine + LLM Client，不新建 AI 通道 |

## 3. 用户场景

### 3.1 目标用户

| 角色 | 场景 | 特点 |
|---|---|---|
| 实施工程师 | 部署后验证数据、排查问题 | 会写 SQL，需要高效编辑体验 |
| 运维人员 | 临时查数据、核对指标 | SQL 能力有限，需要可视化或 AI 辅助 |
| 管理员 | 查看业务数据概况 | 偶尔用，希望快速上手 |

### 3.2 典型场景

**场景 A：实施工程师排查数据问题**
> 收到用户反馈"某天数据不对"，需要快速查询验证。
> 流程：选数据源 → 手写 SQL → 执行 → 查看结果 → 调整 SQL 再执行

**场景 B：运维人员查数据但不会写 SQL**
> 想看"最近 7 天的订单表数据，只看状态为已完成的"。
> 流程（可视化）：选数据源 → 选表 → 选字段 → 加条件 → 生成 SQL → 执行
> 流程（AI）：选数据源 → 选表 → 输入"查询最近7天状态为已完成的订单" → 生成 SQL → 确认执行

**场景 C：复用历史查询**
> 每周都要跑同一个统计查询，不想每次重新写。
> 流程：打开历史记录 → 选择上次的查询 → 填入 SQL 框 → 执行

| 不做的事 | 原因 |

| 完整的 IDE 级代码编辑器 | 产品定位是轻量工具，不引入 Monaco Editor 等重量级依赖 |
| 查询计划可视化 | EXPLAIN 结果已可查看文本，可视化 ROI 不高 |

## 5. 各功能详细设计

### 5.1 SQL 编辑器增强

**当前问题：**
- 纯 `<textarea>` 无语法高亮，SQL 关键字与普通文本无区别
- 无行号
- 暗色背景体验尚可，但缺乏代码编辑器的基本功能

**增强方案：**

采用**轻量级语法高亮**方案，不引入 Monaco Editor（打包体积约 5MB+），而是使用 CSS + 正则覆盖层实现关键字着色。

方案选型：使用 `react-simple-code-editor` + `prismjs`（SQL 语法）的组合。

- 打包体积增加约 30KB（vs Monaco 的 5MB+）
- 支持 SQL 关键字高亮（SELECT、FROM、WHERE、JOIN 等）
- 支持行号显示
- 保持 Ctrl+Enter 执行快捷键
- 保持暗色主题

```
┌─ SQL 编辑器 ──────────────────────────────────────────┐
│  1 │ SELECT id, name, status, update_time             │
│  2 │ FROM result_daily                                 │
│  3 │ WHERE status = '已完成'                           │
│  4 │   AND update_time >= '2026-03-22'                │
│  5 │ ORDER BY update_time DESC                         │
│  6 │ LIMIT 100                                         │
│    │                                                   │
└────────────────────────────────────────────────────────┘
  [▶ 执行 (Ctrl+Enter)]  [📥 导出Excel]  [📥 导出CSV]
```

**SQL 关键字高亮色表（暗色主题）：**
- 保留字（SELECT, FROM, WHERE, JOIN...）：蓝色 `#89b4fa`
- 函数（COUNT, SUM, MAX...）：紫色 `#cba6f7`
- 字符串（'...'）：绿色 `#a6e3a1`
- 数字：橙色 `#fab387`
- 注释（--）：灰色 `#6c7086`

### 5.2 可视化查询构建器（新增）

**用户价值：**
不会写 SQL 的运维人员也能构建查询条件。

**交互流程：**

```
[模式切换: 手写SQL | 可视化构建 | AI生成]

┌─ 可视化查询构建器 ──────────────────────────────────────┐
│                                                          │
│  数据源: [MySQL-生产库 ▼]    表: [result_daily ▼]       │
│                                                          │
│  ── 选择字段 ──                                          │
│  [☑ id] [☑ name] [☑ status] [☑ value] [☐ remark]      │
│  [全选] [反选]                                           │
│                                                          │
│  ── 筛选条件 ──                                          │
│  [status ▼] [= ▼] [已完成    ] [+添加] [×]             │
│  [value  ▼] [> ▼] [100       ] [     ] [×]             │
│                                                          │
│  ── 排序 ──                                              │
│  [update_time ▼] [降序 ▼] [+添加排序]                   │
│                                                          │
│  ── 分组 (可选) ──                                       │
│  [展开]                                                  │
│                                                          │
│  ── 限制行数 ──                                          │
│  [1000 ▼]                                                │
│                                                          │
│  [生成 SQL]  [直接执行]                                  │
└──────────────────────────────────────────────────────────┘
```

**点击"生成 SQL"后：**
- SQL 自动填入编辑器区域
- 用户可在编辑器中微调
- 再点执行

**构建器生成规则：**

| 组件 | 生成的 SQL 部分 |
|---|---|
| 选择字段 | `SELECT field1, field2, ...` （未选则 `SELECT *`） |
| 筛选条件 | `WHERE field op value AND ...` |
| 排序 | `ORDER BY field ASC/DESC` |
| 分组 | `GROUP BY field`（选择分组后，SELECT 自动添加聚合函数选择） |
| 限制行数 | `LIMIT n` |

**筛选条件操作符：**

| 操作符 | 显示 | 适用类型 |
|---|---|---|
| = | 等于 | 全部 |
| != | 不等于 | 全部 |
| > | 大于 | 数值、日期 |
| >= | 大于等于 | 数值、日期 |
| < | 小于 | 数值、日期 |
| <= | 小于等于 | 数值、日期 |
| LIKE | 包含 | 文本 |
| NOT LIKE | 不包含 | 文本 |
| IS NULL | 为空 | 全部 |
| IS NOT NULL | 不为空 | 全部 |
| IN | 在列表中 | 全部（输入逗号分隔值） |
| BETWEEN | 介于 | 数值、日期（两个输入框） |

**后端接口需求：**

构建器需要获取指定数据源中表的字段列表。复用已有的库表管理接口：
- `GET /api/db-manager/tables` — 获取表列表
- `GET /api/db-manager/table-structure` — 获取字段列表（名称 + 类型，用于自动匹配操作符）

SQL 生成在**前端完成**（纯字符串拼接），最终通过 `POST /api/sql-console/execute` 执行。不需要新增后端接口。

**安全说明：**
- 构建器只能生成 SELECT 语句
- 后端依然执行 `_validate_sql` 校验，作为最后一道防线
- 字段名和表名通过下拉选择而非自由输入，降低注入风险

### 5.3 AI 自然语言生成 SQL（新增）

**与已有 NL Query 的区别：**

| 维度 | 已有 NL Query | 新增 AI SQL 生成 |
|---|---|---|
| 入口 | 纳管表数据浏览页面 | SQL 操作台 |
| 范围 | 仅限纳管表 | 任意数据源任意表 |
| 输出 | 结构化 filter 条件 | 完整 SELECT SQL |
| AI 依赖 | 可降级到规则引擎 | 必须有 LLM（无 AI 时隐藏此功能） |
| 安全 | 参数化查询 | 生成 SQL 仍需 `_validate_sql` 校验 |

**交互流程：**

```
[模式切换: 手写SQL | 可视化构建 | AI生成]

┌─ AI 自然语言查询 ──────────────────────────────────────┐
│                                                          │
│  数据源: [MySQL-生产库 ▼]    表: [result_daily ▼]       │
│                                                          │
│  ┌──────────────────────────────────────────────────┐   │
│  │ 查询最近7天状态为已完成的记录，按更新时间倒序     │   │
│  └──────────────────────────────────────────────────┘   │
│  [🤖 生成 SQL]                                          │
│                                                          │
│  ── AI 生成结果 ──                                       │
│  SQL:                                                    │
│  SELECT * FROM result_daily                              │
│  WHERE status = '已完成'                                 │
│    AND update_time >= '2026-03-22'                       │
│  ORDER BY update_time DESC                               │
│                                                          │
│  解释: 查询 result_daily 表中最近7天且状态为已完成的     │
│  全部记录，按更新时间从新到旧排列                        │
│                                                          │
│  [✏️ 编辑SQL]  [▶ 直接执行]                             │
└──────────────────────────────────────────────────────────┘
```

**流程：**
1. 用户选择数据源和表
2. 输入自然语言描述
3. 前端调用 `POST /api/sql-console/ai-generate`
4. 后端获取表的字段信息，构建 prompt 发给 LLM
5. LLM 返回 SQL + 解释
6. 后端对生成的 SQL 执行 `_validate_sql` 校验
7. 前端展示生成的 SQL + 解释
8. 用户可选择：直接执行 / 编辑后执行

**后端接口（新增）：**

```
POST /api/sql-console/ai-generate
{
  "datasource_id": 1,
  "table_name": "result_daily",
  "query_text": "查询最近7天状态为已完成的记录，按更新时间倒序",
  "schema_name": null  // 可选
}

Response:
{
  "sql": "SELECT * FROM result_daily WHERE ...",
  "explanation": "查询 result_daily 表...",
  "confidence": 0.9,
  "engine": "llm"
}
```

**LLM Prompt 设计要点：**
- System Prompt 明确约束：只生成 SELECT 语句，不包含 INSERT/UPDATE/DELETE 等
- 传入表的字段列表（名称 + 类型）作为 context
- 传入当前日期（用于"最近X天"类查询）
- 传入数据库类型（MySQL/PostgreSQL 等，影响方言）
- 要求返回 JSON：`{ sql, explanation, confidence }`

**AI 不可用时的降级：**
- AIEngine 未启用或 LLM Client 不可用时，隐藏"AI 生成"Tab
- 只展示"手写 SQL"和"可视化构建"两种模式

**安全控制：**
- AI 生成的 SQL 必须通过 `_validate_sql` 校验（只允许 SELECT）
- 如果 AI 生成了不安全的 SQL，后端拒绝并返回错误提示
- 前端展示生成的 SQL，用户确认后才执行（不自动执行）

### 5.4 查询历史（新增）

**存储方案：**
前端 localStorage 存储，不增加后端复杂度。

```typescript
interface QueryHistoryItem {
  id: string;           // uuid
  sql: string;
  datasource_id: number;
  datasource_name: string;
  executed_at: string;  // ISO datetime
  row_count?: number;
  success: boolean;
  error?: string;
}
```

**交互：**
- 在 SQL 编辑器下方显示"历史记录"折叠面板
- 展示最近 50 条查询（倒序）
- 每条显示：SQL 前 80 字符 + 数据源名称 + 执行时间 + 行数
- 点击某条：将 SQL 填入编辑器，切换到对应数据源
- 支持清空历史
- 成功和失败的查询都记录（失败的标红）

```
┌─ 查询历史 ──────────────────────────────────────────────┐
```

## 7. 与现有功能的关系

### 7.1 与 NL Query 的关系

```
┌─ 纳管表数据浏览页 ─┐     ┌─ SQL 操作台 ──────────────┐
│                      │     │                            │
│  NL Query            │     │  AI SQL 生成               │
│  (结构化 filter)     │     │  (任意 SELECT SQL)         │
│                      │     │                            │
│  ↓ 共用               │     │  ↓ 共用                    │
│  AIEngine + LLM      │     │  AIEngine + LLM            │
│                      │     │                            │
│  适合：纳管表内的     │     │  适合：任意表的            │
│  数据筛选浏览        │     │  灵活查询探索              │
└──────────────────────┘     └────────────────────────────┘
```

两者共用 AIEngine 和 LLM Client，但 Prompt 不同：
- NL Query：输出结构化 filter，后端拼装参数化 SQL（更安全）
- AI SQL 生成：输出完整 SQL（更灵活，但需要 `_validate_sql` 兜底）

**不合并的原因：**
- 使用场景不同（日常数据浏览 vs 灵活探索查询）
- 安全模型不同（参数化 vs SQL 校验）
- 依赖不同（纳管表配置 vs 原始表结构）

### 7.2 与库表管理的关系

- SQL 操作台的**可视化构建器**需要调用库表管理的接口获取表列表和字段列表
- 两者共享数据源选择器的逻辑
- SQL 操作台只读，库表管理可写（DDL）

### 7.3 与数据回写的关系

- SQL 操作台不参与数据回写流程
- 可以用 SQL 操作台查询验证回写结果

## 8. 安全设计

### 8.1 只读原则

SQL 操作台**坚持只允许 SELECT 查询**，理由：
1. 产品定位是查询工具，写操作有专门的模块负责
2. 写操作需要审批流程和回退能力，SQL 操作台不具备
3. AI 生成的 SQL 如果包含写操作，风险不可控

### 8.2 安全防线

| 防线 | 说明 |
|---|---|
| 前端：可视化构建器只能生成 SELECT | 从源头杜绝危险 SQL |
| 前端：AI 生成后展示预览，用户确认 | 人工审核 AI 输出 |
| 后端：`_validate_sql` 白名单校验 | 只允许 SELECT/WITH/EXPLAIN/SHOW/DESCRIBE |
| 后端：`BLOCKED_KEYWORDS` 黑名单 | 阻止 INSERT/UPDATE/DELETE/DROP 等 |
| 后端：多语句禁止 | 防止 `SELECT 1; DROP TABLE` 攻击 |
| 后端：`max_rows` 限制 | 防止 `SELECT *` 拖垮大表 |
| AI Prompt：明确约束只生成 SELECT | 降低 AI 生成危险 SQL 的概率 |

### 8.3 AI SQL 注入防范

AI 生成的 SQL 是**不可信输入**，必须经过与手写 SQL 相同的安全校验。额外措施：
- Prompt 中明确告知 LLM "你只能生成 SELECT 语句"
- 后端对 AI 返回的 SQL 执行 `_validate_sql`，不通过则返回错误
- 记录 AI 生成的原始 SQL 到审计日志（包括被拒绝的）

### 8.4 权限

| 操作 | 所需角色 |
|---|---|
| 手写 SQL 执行 | admin |
| 可视化构建查询 | admin（远期可降至 operator） |
| AI 生成 SQL | admin |
| 导出结果 | admin |
| 查看历史 | 本地存储，无权限控制 |

## 12. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| prismjs 打包体积 | 前端包增大 | 只引入 SQL 语言包，tree-shake 其余 |
| AI 生成不安全 SQL | 执行写操作 | 后端 `_validate_sql` 强制校验，AI Prompt 约束 |
| AI 生成的 SQL 语法错误 | 执行失败 | 前端展示错误信息，用户可手动修改 |
| 可视化构建器字段列表加载慢 | 大库体验差 | 复用库表管理缓存，加 loading 状态 |
| LLM 服务不可用 | AI 功能瘫痪 | 优雅降级，隐藏 AI Tab，不影响手写和可视化 |
| localStorage 历史记录过大 | 浏览器存储告警 | 限制 50 条，SQL 截断至 2000 字符 |
