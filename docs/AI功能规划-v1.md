# 数据运维工作台 - AI 功能规划

> 版本：v1  
> Owner：斯维因（产品经理）  
> 日期：2026-03-24  
> 定位：基于政企数据运维场景，规划 AI 在工作台中的应用方向与落地优先级

---

## 一、背景与原则

数据运维工作台服务的是政企项目交付和运维现场。这类场景的特点：
- 操作人员非技术背景，怕出错、怕背锅
- 数据修改频繁但每次改动量不大，重复性高
- 内网部署为主，对外部 API 调用有严格限制
- 数据敏感，不允许数据出域

**AI 规划原则：**
1. **辅助而非替代** — AI 做建议和校验，最终决策权在人
2. **可离线运行** — 优先支持本地小模型，不强依赖云端 API
3. **渐进引入** — 从低风险的只读辅助开始，逐步扩展到写入辅助
4. **透明可解释** — AI 的每个建议都要给出理由，用户能看懂为什么

## 二、场景与需求规划

### 场景 1：智能字段配置助手

**痛点：** 管理员配置纳管表时，需要逐个字段设置语义名称、是否可编辑、是否必填、枚举值等。字段多的表（50+ 字段）配置耗时且容易遗漏。

**AI 能力：**
- 根据字段名、数据类型、样本数据，自动推荐：
  - 中文语义名（如 `create_time` → 创建时间）
  - 是否应设为只读（如时间戳字段、自增 ID）
  - 是否为系统字段（如 `created_by`、`updated_at`）
  - 可能的枚举值（如 `status` 字段采样后推荐 "正常/停用/待审核"）
- 管理员可一键采纳或逐项调整

**价值：** 50 个字段的表，配置时间从 20 分钟降到 3 分钟

**风险等级：** 低 — 只是配置建议，不碰业务数据

**优先级：⭐⭐⭐⭐⭐ 推荐 v2.0 首批落地**

---

### 场景 2：导入数据智能校验增强

**痛点：** 当前校验规则是基于配置的硬规则（类型、长度、必填）。实际运维中，很多错误是"格式对但内容不合理"，硬规则捕捉不了。

**AI 能力：**
- **异常值检测：** 基于历史数据分布，标记偏离正常范围的值
  - 例：某数值字段历史值在 0-100 之间，用户填了 99999 → 标黄提醒
- **格式一致性检查：** 检测同一字段内格式不统一
  - 例：手机号字段混入了座机号、带区号的号码
- **重复值预警：** 非主键字段的异常重复（如姓名字段全部变成同一个值）
- **关联合理性：** 简单的跨字段逻辑检查
  - 例：开始日期 > 结束日期 → 提醒

**价值：** 减少"写进去才发现数据有问题"的事后回退

**风险等级：** 低 — 只做提醒，不阻断操作（除非用户配置为阻断）

**优先级：⭐⭐⭐⭐ 推荐 v2.0 落地**

---

### 场景 3：自然语言数据查询

**痛点：** 当前数据浏览只支持关键字和字段筛选。运维人员有时需要更灵活的查询，比如"找出上个月修改过的所有状态异常的记录"，但他们不会写 SQL。

**AI 能力：**
- 用户用自然语言描述查询意图
- AI 转换为结构化筛选条件（不直接生成 SQL，而是转为平台已有的筛选参数）
- 显示 AI 理解的筛选条件，用户确认后执行
- 支持追问式优化："再加一个条件，只看北京地区的"

**示例：**
- 用户输入："找出最近 7 天更新过的、状态不是正常的记录"
- AI 输出筛选条件：更新时间 ≥ 2026-03-17，状态 ≠ 正常
- 用户确认 → 执行查询

**价值：** 降低数据查询门槛，非技术人员也能灵活筛数据

**风险等级：** 低 — 只读操作，且有确认环节

**优先级：⭐⭐⭐ 推荐 v2.1 落地**

---

### 场景 4：操作日志智能分析

**痛点：** 日志中心目前只是流水账式的日志列表。当出现数据问题需要溯源时，管理员要手动翻大量日志，效率低。

**AI 能力：**
- **操作摘要：** 对指定时间段的操作日志生成自然语言摘要
  - "过去 24 小时，用户张三对【客户信息表】执行了 3 次回写，修改了 47 条记录，其中 2 次触发了校验警告"
- **异常操作识别：** 标记不寻常的操作模式
  - 某用户短时间内对同一张表反复回写
  - 某次回写修改了异常多的字段或记录
  - 非工作时间的操作
- **溯源辅助：** "这条记录的【状态】字段最近一次是谁改的？改之前是什么值？" → AI 从日志中定位回答

**价值：** 问题溯源从"翻半小时日志"变成"问一句话"

**风险等级：** 低 — 只读分析

**优先级：⭐⭐⭐ 推荐 v2.1 落地**

---

### 场景 5：Excel 模板智能填充助手

**痛点：** 运维人员下载模板后，有时需要批量修改数据但修改规则是有规律的（比如"所有部门为A的记录，状态改为停用"），只能一行行手动改。

**AI 能力：**
- 用户在上传模板前，可以用自然语言描述批量修改意图
- AI 生成修改预览（哪些行的哪些字段会被改成什么值）
- 用户确认后，AI 自动填充到模板数据中
- 再走正常的导入 → 校验 → 差异预览 → 回写流程

**示例：**
- 用户描述："把所有部门是'华北区'的记录，负责人改成'李明'"
- AI 预览：将修改第 3、7、12、15 行的【负责人】字段，从原值改为"李明"
- 用户确认 → 生成修改后的数据 → 进入校验流程

**价值：** 批量规律性修改效率提升 10 倍

**风险等级：** 中 — 涉及数据修改，但有差异预览和回写确认兜底

**优先级：⭐⭐⭐ 推荐 v2.1 落地**

---

### 场景 6：数据源健康巡检

**痛点：** 政企现场经常出现数据库连接异常、表结构被人改了、磁盘空间不够等问题，发现时往往已经影响业务。

**AI 能力：**
- 定时巡检各数据源连接状态
- 检测表结构变化并主动提醒
- 基于历史数据增长趋势，预警可能的容量问题
- 生成巡检报告（可导出）

**价值：** 变被动运维为主动运维，减少故障响应时间

**风险等级：** 低 — 只读巡检

**优先级：⭐⭐ 推荐 v2.2 或更后**

---

### 场景 7：回写影响预评估

**痛点：** 用户提交回写前，差异预览只显示"哪些字段变了"。但在政企场景中，某些字段的修改可能有下游影响（比如改了状态字段，可能影响其他系统的数据同步）。

**AI 能力：**
- 回写前，AI 基于字段元数据和历史回写记录，评估本次修改的影响面
- 标记高敏感字段的修改（管理员可配置哪些字段为高敏感）
- 给出风险等级提示："本次回写涉及 3 个高敏感字段的修改，建议审核后再执行"

**价值：** 减少误操作导致的连锁问题

**风险等级：** 低 — 只做评估提示

**优先级：⭐⭐ 推荐 v2.2 或更后**

## 三、技术实现方向

考虑到政企内网部署的刚性约束：

| 方案 | 说明 | 适用场景 |
|---|---|---|
| 本地小模型 | 内嵌轻量模型（如 Qwen-1.5B / GLM-Edge），随工作台一起部署 | 字段识别、异常检测、简单 NL 查询 |
| 规则+统计 | 不用大模型，基于统计分析和规则引擎实现 | 异常值检测、日志分析、巡检 |
| 可选云端 API | 支持配置外部 AI 接口（如企业私有化部署的大模型），但不强依赖 | 自然语言查询、复杂分析 |

**推荐策略：** 核心 AI 功能用"规则+统计+本地小模型"兜底保证离线可用，同时预留云端 API 接口，有条件的环境可以接入更强的模型获得更好体验。

## 四、落地路线

```
v2.0  ── 智能字段配置助手 + 导入数据智能校验增强
  │      （配置提效 + 数据质量提升，低风险高回报）
  │
v2.1  ── 自然语言查询 + 日志智能分析 + 模板智能填充
  │      （操作提效，中等复杂度）
  │
v2.2  ── 数据源巡检 + 回写影响评估
         （运维增强，需要更多数据积累）
```
