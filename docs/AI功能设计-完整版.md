
后端统一封装 `AIClient` 类，根据协议格式选择对应的请求方式，业务代码无感知。

#### 本地模型效果补偿策略

考虑到政企内网环境可能只能用本地小模型（1.5B-7B 参数），平台层面设计了 7 层补偿机制确保效果可用：

| 策略 | 机制 | 效果 |
|---|---|---|
| **规则兜底** | 80% 任务用内置规则引擎完成，AI 只做增量 | 即使 AI 不可用，系统仍提供 80% 辅助能力 |
| **精准 Prompt** | 受限 JSON 格式输出 + 精简上下文 + few-shot | 降低对模型能力的要求 |
| **置信度阈值** | ≥0.8 直接展示、0.5-0.8 标黄提醒、<0.5 不展示 | 宁可不推荐也不给错误推荐 |
| **用户反馈回流** | 采纳/拒绝记录回流到规则引擎 | 用得越多越准 |
| **任务拆解** | 复合任务拆为多步，AI 只做最核心语义理解 | 每步简单，小模型也能做 |
| **能力分级提示** | 根据引擎模式提示可用功能范围 | 管理用户预期 |
| **结果验证** | AI 输出过验证层（枚举比对/约束检查/dry-run） | 拦住 AI 的错误输出 |

详细说明：
- **规则兜底：** 字段推荐有内置映射表（覆盖常见字段 80%+），异常值检测用 P5-P95 分位值（纯统计），NL 查询用关键词匹配（覆盖常见查询模式）
- **渐进式学习：** 用户每次采纳/拒绝推荐，系统记录到本地知识库。随着使用积累，规则引擎的覆盖率逐步提升，对 AI 模型的依赖逐步降低
- **能力分级：** 管理员在设置页面可以看到当前引擎模式下哪些 AI 功能可用、哪些降级为规则模式

### 1.3 交互流程

```
管理员进入字段配置页
  │
  ├─ 点击「AI 智能推荐」
  │   │
  │   ├─ 前端调 POST /api/ai/field-suggest
  │   │   请求体：{ table_id, sample_count: 100 }
  │   │
  │   ├─ 后端：
  │   │   1. 从目标库采样 sample_count 条数据
  │   │   2. 对每个字段分析：字段名 → 语义名映射
  │   │   3. 根据数据类型 + 字段名模式 → 推荐只读/系统字段
  │   │   4. 对非数值字段去重 → 推荐枚举值（唯一值 ≤ 20 个时）
  │   │   5. 对数值字段计算 min/max/avg → 判断范围
  │   │   6. 返回推荐列表
  │   │
  │   ├─ 前端展示推荐弹窗
  │   │
  │   ├─ 管理员可：
  │   │   - 逐条采纳（点击某条的[采纳]）
  │   │   - 全部采纳
  │   │   - 仅采纳语义名（只应用中文名，不改配置）
  │   │   - 修改推荐值后再采纳
  │   │   - 取消忽略
  │   │
  │   └─ 采纳后，字段配置表实时更新，但不自动保存
  │
  └─ 管理员确认无误后点击「保存配置」
```

### 1.5 内置规则引擎逻辑

不依赖 LLM 即可工作的规则：

**字段名 → 语义名映射表：**
```
id / pk → 编号
name / user_name / username → 用户名
status / state → 状态
create_time / created_at / gmt_create → 创建时间
update_time / updated_at / gmt_modified → 更新时间
create_by / created_by / creator → 创建人
update_by / updated_by / modifier → 更新人
phone / mobile / tel → 手机号/电话
email / mail → 邮箱
address / addr → 地址
remark / memo / note / comment → 备注
dept / department / org → 部门
type / category → 类型
code / no / number → 编码
is_deleted / deleted / del_flag → 删除标记
sort / order_num / seq → 排序号
```
（支持配置扩展）

**只读推荐规则：**
- 字段名匹配 `id|pk|create_time|created_at|update_time|updated_at|create_by|update_by|is_deleted|del_flag`
- 数据类型为 AUTO_INCREMENT / SERIAL
- 字段为表主键

**系统字段推荐规则：**
- 字段名匹配 `create_*|update_*|gmt_*|is_deleted|del_flag|version|revision`

**枚举值推荐规则：**
- 非数值字段
- 采样数据中唯一值 ≤ 20 个
- 返回去重后的值列表，按出现频率排序

### 1.6 LLM 增强（可选）

当配置了 LLM 时，额外能力：
- 更准确的语义名推荐（理解缩写、行业术语）
- 字段间关联关系推断（如 `start_date` 和 `end_date` 是一对）
- 业务含义描述生成（如"该字段记录用户账号的启用/禁用状态"）

### 1.7 验收标准

- [ ] 点击「AI 智能推荐」后 3 秒内返回结果（100 条采样）
- [ ] 规则引擎模式下，无需任何外部依赖即可工作
- [ ] 推荐结果每条都有 reason 字段，用户可理解
- [ ] 逐条采纳和全部采纳功能正常
- [ ] 采纳后字段配置表实时更新

- [ ] 无字段匹配规则时，显示"无推荐"而非报错

## 二、导入数据智能校验增强（v2.0）

### 2.1 功能定位

在现有硬规则校验（类型、长度、必填）基础上，增加基于数据统计分析的智能校验，帮用户在回写前发现"格式对但内容不合理"的数据问题。

### 2.3 智能校验规则

#### 2.3.1 异常值检测

| 字段类型 | 检测逻辑 | 示例 |
|---|---|---|
| 数值字段 | 计算历史数据的 P5-P95 分位值，超出范围标警告 | 年龄历史 18-65，新值 250 |
| 文本字段 | 检测文本长度异常（远超/远短于历史平均） | 姓名字段出现 200 字长文本 |

#### 2.3.2 格式一致性检测

| 检测项 | 逻辑 | 示例 |
|---|---|---|
| 手机号/电话 | 正则匹配历史主流格式，不一致标警告 | 混入座机号 |

| 编码字段 | 前缀/长度不一致 | 编码都是 `BJ-` 开头，新值是 `SH-` |

#### 2.3.3 重复值预警

| 检测项 | 逻辑 | 示例 |
|---|---|---|
| 全列相同值 | 非主键字段，导入数据某列全部是同一个值 | 批量修改后所有部门变成同一个 |
| 唯一性字段重复 | 历史数据唯一的字段出现重复 | 手机号字段出现重复值 |

#### 2.3.4 跨字段逻辑检查

| 检测项 | 逻辑 | 示例 |
|---|---|---|
| 日期先后关系 | 开始日期 > 结束日期 | start_date 晚于 end_date |
| 数值关系 | 最小值 > 最大值 | min_price > max_price |

### 3.5 NL 解析策略

**内置规则模式（无需 LLM）：**
- 关键词匹配："最近X天" → 日期范围
- 否定词识别："不是/不等于/排除" → neq 操作符
- 字段名模糊匹配：用户说"更新过的" → 匹配 update_time 字段
- 枚举值匹配：用户说"停用的" → 匹配 status 字段的"停用"值
- 数值比较："大于100/超过100" → gt 操作符

**LLM 增强模式：**
- 理解更复杂的表述
- 支持追问和上下文关联
- 处理模糊语义（"异常的记录" → 可能指状态异常，也可能指数值异常）

### 3.6 验收标准

- [ ] 内置规则模式下，常见查询（时间范围、等于/不等于、大于/小于）解析成功率 ≥ 80%
- [ ] 解析结果以人话展示，用户能理解每个筛选条件
- [ ] 用户确认后才执行查询，不自动查
- [ ] 解析失败时提示"无法理解，请换个方式描述"，不报错
- [ ] 支持追问补充条件
- [ ] 查询结果复用现有数据浏览组件，无需重复开发

| 评估维度 | 规则 | 风险标记 |

### 7.5 验收标准

- [ ] 管理员可配置每个字段的敏感等级和影响说明

### 10.5 交互流程

#### 备份流程
```
用户进入"数据备份与迁移"页面
  │
  ├─ 选择备份内容（配置必选，日志和历史备份可选）
  │
  ├─ 点击「立即备份」
  │   │
  │   ├─ 后端：
  │   │   1. 锁定平台数据库（短暂只读）
  │   │   2. 导出 SQLite 数据库拷贝
  │   │   3. 导出各配置为 JSON（脱敏处理）
  │   │   4. 生成 manifest.json
  │   │   5. 打包为 ZIP
  │   │   6. 解除锁定
  │   │
  │   └─ 返回下载链接
  │
  └─ 用户下载 ZIP 文件保存
```

#### 导入流程
```
用户上传备份 ZIP 文件
  │
  ├─ 后端解析 manifest.json，展示备份信息
  │
  ├─ 用户选择导入模式（完全覆盖 / 合并导入）
  │
  ├─ 用户点击「确认导入」
  │   │
  │   ├─ 二次确认弹窗
  │   │
  │   ├─ 后端：
  │   │   1. 自动创建当前数据的临时备份
  │   │   2. 校验备份文件完整性
  │   │   3. 校验版本兼容性
  │   │   4. 执行导入（覆盖或合并）
  │   │   5. 重新加载配置
  │   │
  │   └─ 返回导入结果
  │
  ├─ 导入成功 → 提示"导入完成，建议重新登录"
  │
  └─ 导入失败 → 自动回滚到临时备份，提示失败原因
```

### 10.7 安全设计

| 安全项 | 处理方式 |
|---|---|
| 数据源密码 | 备份时脱敏为 `***`，导入后需用户重新填写 |
| API Key | 备份时脱敏为 `***`，导入后需重新配置 |
| 用户密码 | 导出为哈希值，导入后可直接使用（同算法） |
| 文件完整性 | manifest 中包含文件校验和（SHA256） |
| 版本兼容 | 检查 app_version，不兼容时阻断并提示 |
| 导入保护 | 导入前自动创建当前数据临时备份，失败可回滚 |
