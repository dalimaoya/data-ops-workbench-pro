# 数据运维工作台 PRD v1

## 1. 产品名称

**数据运维工作台**  
副标题：**面向交付与运维场景的数据库结果数据安全可视化维护工具**

---

## 2. 产品定位

数据运维工作台不是数据库开发工具，也不是传统数据填报平台。  
它的定位是：

**围绕数据库中的结果数据表，提供“读取、模板导出、Excel 修订、校验预览、安全回写、留痕回退”的轻量工作台。**

它要替代的是以下几种低效且高风险的方式：

- 直接用 Navicat / SQL 改库
- 线下 Excel 往返传递，人工复制粘贴回库
- 口头通知研发改数
- 运维靠脚本临时处理，过程不可见

---

## 3. 产品目标

### 3.1 核心目标
让普通运维人员或业务维护人员，在**不直接接触数据库**的前提下，完成结果数据的安全更新。

### 3.2 次级目标
把研发的一次性配置，转换成后续普通用户可重复执行的标准流程。

### 3.3 沉淀目标
形成一个可复用、可演示、可开源外壳化的工作台成果物，用于：

- 项目交付提效
- 运维流程规范化
- 个人方法论沉淀
- 案例展示与品牌输出

---

## 4. 非目标

本产品 **不做** 以下事项：

- 不做数据库建模工具
- 不做 SQL 开发平台
- 不做 BI 分析展示平台
- 不做复杂审批流平台
- 不做任意 Excel 自由导入系统
- 不做业务指标计算引擎
- 不做主数据治理平台

---

## 5. 目标用户

### 5.1 配置角色
**研发 / 实施工程师 / 管理员**

职责：
- 配置数据库连接
- 配置表与字段语义
- 指定可编辑字段
- 指定主键/唯一识别字段
- 处理表结构变化

### 5.2 使用角色
**普通运维人员 / 业务维护人员 / 项目管理员**

职责：
- 查看表数据
- 下载模板
- 修改模板
- 上传回填
- 查看差异与报错
- 确认回写

### 5.3 管理角色
**项目负责人 / 高级管理员**

职责：
- 查看系统日志
- 查看版本回退记录
- 执行版本回退
- 管理配置状态

## 6. 核心使用场景

### 场景 1：结果表日常修订
用户打开某张结果表，下载当前数据模板，修改后上传，系统校验通过后回写数据库。

### 场景 2：专题数据临时调整
某专题页面数据异常，运维人员无需找研发写 SQL，通过工作台导出模板后修订指定记录并回写。

### 场景 3：领导汇报前批量修正
用户针对多个指标做集中修订，通过模板批量修改后上传，系统展示差异预览后统一写入。

### 场景 4：结构变化后的重新配置
项目迭代中表新增字段，管理员在配置页补充新字段定义，普通用户继续按原流程使用。

## 7. 产品形态

### 7.1 推荐形态
**本地 exe + 内嵌网页工作台 + 本地服务**

### 7.2 部署方式

### 7.3 不建议形态

---

## 8. 功能边界

### 8.1 产品只做什么
- 连接指定数据库
- 读取指定结果表
- 展示表数据
- 导出平台受控模板
- 导入修订模板
- 执行校验与差异预览
- 执行安全回写
- 记录日志与回退版本

### 8.2 产品不做什么
- 不允许用户直接执行 SQL
- 不允许自由编辑任意数据库对象
- 不允许上传自定义格式 Excel
- 不允许绕过平台直接写入

---

## 9. 核心流程

### 9.1 配置流程（管理员）
1. 新建数据源
2. 选择数据库类型
3. 填写连接信息
4. 获取库表列表
5. 选择需纳入维护的表
6. 自动读取字段信息与样例数据
7. 配置字段语义与编辑规则
8. 保存配置版本

### 9.2 用户使用流程
1. 选择目标表
2. 查看当前数据
3. 选择“导出当前数据”或“导出全量数据”
4. 下载平台模板
5. 本地修改模板
6. 上传模板
7. 系统校验并生成差异预览
8. 用户确认写入
9. 系统备份并回写
10. 生成日志与版本记录

---

## 10. 核心功能模块

### 10.1 数据源管理
功能：
- 新增/编辑/删除数据源
- 支持多数据库类型
- 测试连接
- 获取库表清单

首版建议支持：
- MySQL
- PostgreSQL
- SQL Server

后续可扩：
- Oracle
- 达梦
- 人大金仓

---

### 10.2 表配置管理
功能：
- 选择纳入维护的表
- 记录库名/schema/表名
- 自动拉取字段名、字段类型、样例数据
- 配置字段含义
- 配置字段是否可编辑
- 配置字段是否展示
- 配置字段是否必填
- 配置主键/唯一识别字段
- 配置配置版本号

说明：  
**表唯一标识不以表名单独判定**，而以  
`数据源 + 库/Schema + 表名 + 配置版本`  
进行识别。

---

### 10.3 数据浏览
功能：
- 表数据分页展示
- 全字段文本显示
- 按关键字筛选
- 按字段筛选
- 当前数据查看
- 全量数据查看

首版要求：
- 所有字段统一按文本展示
- 不做复杂组件渲染

---

### 10.4 模板导出
功能：
- 导出当前筛选结果
- 导出全量数据
- 导出为 Excel 模板

模板要求：
- 所有字段以文本形式输出
- 内含平台元信息
- 非平台模板禁止导入

模板隐藏元信息建议包含：
- 数据源 ID
- 表配置 ID
- 配置版本
- 导出时间
- 导出批次号
- 字段编码
- 主键标识

---

### 10.5 模板导入
功能：
- 上传平台模板
- 校验模板有效性
- 校验字段完整性
- 校验版本是否过期
- 校验数据行合法性

要求：  
**用户不可使用自定义 Excel 模板上传，必须使用平台导出的模板。**

---

### 10.6 导入校验
上传后不直接写库，先进入校验阶段。

校验项包括：
- 模板合法性校验
- 必填字段校验
- 字段数量校验
- 字段顺序校验
- 主键/唯一键校验
- 空值校验
- 重复行校验
- 长度限制校验
- 数据类型转换校验
- 枚举值校验（如有配置）
- 字段结构版本校验

校验结果分为：
- 可成功写入
- 校验失败
- 警告但可继续

---

### 10.7 差异预览
功能：
- 展示原值与新值对比
- 按行查看差异
- 按字段查看差异
- 展示拟新增/拟修改/拟失败数量
- 允许用户确认或取消本次写入

这是主流程核心，不可省略。

---

### 10.8 安全回写
写入流程：
1. 检查当前表结构是否与配置一致
2. 生成本次备份
3. 执行写入
4. 写入成功后记录日志
5. 更新版本记录

首版策略：
- 写入前全表备份
- 最多保留最近 3 个备份版本

后续增强：
- 增加按记录/按批次变更日志
- 支持细粒度回退

---

### 10.9 日志与回退
功能：
- 系统操作日志
- 数据写入日志
- 导入失败日志
- 版本备份列表
- 一键回退指定版本

日志字段建议包括：
- 数据源
- 表名
- 操作类型
- 操作人
- 操作时间
- 批次号
- 成功数
- 失败数
- 备注

---

### 10.10 表结构变化检测
每次进入表或执行导入前，系统自动比对当前库表结构与已保存配置。

检测项：
- 是否新增字段
- 是否删除字段
- 字段类型是否变化
- 主键是否变化

发现变化后：
- 禁止直接写入
- 提示“表结构已变化，请管理员重新确认配置”

---

## 11. 配置策略

### 11.1 首次接入
首次接入时，系统自动拉取：

- 字段名
- 字段类型
- 前 3~10 条样例数据

管理员进行人工确认：
- 该字段含义
- 是否展示
- 是否可编辑
- 是否必填
- 是否为主键
- 是否为系统字段

### 11.2 后续维护
普通用户不维护配置。  
如表字段变化，由研发/管理员补充配置一次即可。

---

## 12. 数据策略

### 12.1 展示策略
- 页面统一按文本形式展示字段值

### 12.2 模板策略
- 模板统一按文本形式导出

### 12.3 写入策略
- 导入后由系统按数据库字段类型做强制转换
- 转换失败则报错，不允许写入

### 12.4 写入范围
首版仅支持：
- 更新已有记录
- 不默认支持自由新增记录
- 不默认支持删除记录

这样能显著降低风险。

## 16. 技术架构建议

### 前端
- React
- Ant Design / Naive UI
- Excel 上传下载组件

### 后端
- Python FastAPI 或 Node.js NestJS
- 建议优先 FastAPI，适合快速落地

### 数据层
- 平台自身数据库：存配置、日志、版本信息
- 外部业务数据库：存被维护的结果表

### 打包方式
- Tauri 优先
- Electron 备选

理由：
- Tauri 更轻
- 更适合做本地 exe

---

## 17. 平台自身数据表建议

建议至少有以下几张表：

- `datasource_config`：数据源配置表
- `table_config`：纳管表配置表
- `field_config`：字段配置表
- `template_export_log`：模板导出日志
- `import_task_log`：导入任务日志
- `writeback_log`：回写日志
- `table_backup_version`：表备份版本记录
- `system_operation_log`：系统操作日志

---

## 18. 成功标准

首版上线后，满足以下标准即算成功：

- 普通用户无需接触数据库客户端即可更新结果表
- 用户必须通过平台模板完成修订
- 平台可识别错误模板并阻止写入
- 每次写入前均有备份
- 出错后可回退到最近版本
- 研发只需首配一次，后续普通用户可独立使用

---

## 19. 产品一句话价值

**用可视化、可控、可回退的方式，替代交付现场“直接改数据库”的高风险维护流程。**
