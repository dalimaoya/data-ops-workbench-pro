## 1. 背景与动机

### 1.1 现状

当前 plugin-db-manager 已具备：
- 表列表浏览（7 种数据库）
- 表结构查看（字段名、类型、主键、可空、默认值、注释）
- 建表（可视化定义字段 + SQL 预览 + 执行）
- 字段管理（添加字段、修改字段、删除字段）
- 索引管理（查看索引、创建索引、删除索引）
- 删表（确认 + 自动备份）
- 全操作审计留痕

### 1.2 用户反馈

- 同事提出需要"类似 Navicat 的数据库管理能力"
- 现场运维人员希望能在工作台内完成简单的表结构调整，不再切换到数据库客户端
- 未来可能需要数据同步功能（跨库迁移、定时同步）

### 1.3 核心矛盾

> 我们不是要做一个 Navicat，而是数据运维场景下的轻量管理工具。

产品定位是"不碰数据库客户端也能安全修订结果表"。库表管理是这个定位的**结构层延伸**——让用户在同一工作台内完成从"表结构准备"到"数据修订"的完整闭环，而非提供通用的数据库开发环境。

## 2. 设计原则

| 原则 | 说明 |
|---|---|
| 运维优先，开发次之 | 服务于数据运维场景（建表、加字段、调索引），不做存储过程、触发器、视图管理 |
| 安全兜底 | DDL 操作必须有预览、确认、备份三重保护；高危操作需二次确认 |
| 渐进式复杂度 | 默认展示简洁视图，高级选项折叠；让普通用户不恐惧，让高级用户够用 |
| SQL 透明 | 所有结构变更都先生成 SQL 预览，用户确认后再执行，培养用户 SQL 认知 |
| 7 库兼容 | MySQL / PostgreSQL / SQL Server / Oracle / 达梦 / 人大金仓 / SQLite 全覆盖 |

## 3. 用户场景

### 3.1 目标用户

| 角色 | 场景 | 频率 |
|---|---|---|
| 实施工程师 | 项目交付时建表、调整表结构，为数据回写做准备 | 项目初期高频，后期低频 |
| 运维人员 | 业务变更后加字段、改字段类型，不想找研发写 SQL | 按需，月均 2-5 次 |
| 管理员 | 查看表结构、索引情况，评估性能问题 | 按需 |
| 研发人员 | 快速查看远程库表结构，辅助排查问题 | 按需 |

### 3.2 典型场景

**场景 A：项目交付建表**
> 实施工程师需要在客户环境创建结果表，然后配置为纳管表供运维人员使用。
> 流程：选数据源 → 建表（定义字段）→ SQL 预览确认 → 执行 → 跳转纳管表配置

**场景 B：业务变更加字段**
> 业务新增一个指标维度，运维人员需要给结果表加一个字段。
> 流程：选表 → 添加字段 → 预览 SQL → 执行 → 刷新纳管表字段配置

**场景 C：性能优化加索引**
> 数据量增长后查询变慢，管理员需要给高频查询字段加索引。
> 流程：选表 → 查看索引 → 创建索引 → 预览 SQL → 执行

**场景 D：表结构迁移（未来）**
> 把 A 库的表结构复制到 B 库，用于多环境部署。
> 流程：选源表 → 导出建表 SQL → 切换目标数据源 → 执行

### 4.3 远期范围（P2/P3）

| 优先级 | 功能 | 说明 |
|---|---|---|
| P2 | 表结构对比 | 选择两个数据源的同名表，对比字段差异 |
| P2 | 批量 DDL | 一次性添加多个字段 |
| P3 | 数据同步 — 一次性迁移 | 将 A 库表数据迁移到 B 库（结构 + 数据） |
| P3 | 数据同步 — 定时同步 | 定时从源库拉取增量数据写入目标库 |

### 4.4 明确不做

| 不做的事 | 原因 |
|---|---|
| 视图管理 | 非运维场景核心需求，复杂度高 |
| 存储过程/函数/触发器 | 属于数据库开发范畴，偏离产品定位 |
| 用户/权限管理（数据库层） | 涉及数据库安全，应由 DBA 在专业工具中操作 |
| 数据库创建/删除 | 超出运维工作台边界 |
| SQL 编辑器（通用） | 已有 AI 自然语言查询覆盖查询场景 |
| 数据库性能监控 | 属于专业 DBA 工具范畴 |

## 5. 各功能详细设计

### 5.1 表浏览增强

**当前问题：**
- 表列表无搜索能力，数据源有几百张表时难以定位
- 无表数量统计
- 无刷新按钮（需切换数据源再切回来）

**增强方案：**

```
┌─ 数据源选择 ─┐  ┌─ 表搜索框 ─┐  [刷新] [建表]
│ MySQL-生产库  │  │ 输入表名... │
└──────────────┘  └────────────┘

共 127 张表，显示 12 张（已过滤）

┌─────────────────────────────────────┐
│ 表名              │ 行数(估) │ 注释  │
├─────────────────────────────────────┤
│ result_daily      │ ~12,000  │ 日报表│
│ result_monthly    │ ~3,200   │ 月报表│
│ ...               │          │       │
└─────────────────────────────────────┘
```

**交互要点：**
- 表列表改为左侧面板 + 右侧详情的分栏布局（参考 Navicat 的树形结构，但更简洁）
- 表名搜索框支持模糊匹配，输入即过滤
- 显示表总数和过滤后数量
- 提供手动刷新按钮

**后端接口变更：**
- `GET /api/db-manager/tables` 增加可选参数 `keyword` 用于服务端过滤
- 返回增加 `row_count_estimate`（仅 MySQL/PostgreSQL 支持，其他库返回 null）

### 5.2 数据预览（新增）

**用户价值：**
选中表后快速查看数据内容，确认表是否是自己要操作的目标表。

**交互流程：**
1. 选中表后，在详情区增加"数据预览"Tab（与"表结构""索引"并列）
2. 默认显示前 100 行
3. 支持排序（点击列头）
4. 只读，不可编辑
5. 显示总行数（估算值）

```
Tabs: [表结构] [索引] [数据预览]

┌────────────────────────────────────────────┐
│ id │ name     │ value │ update_time        │
├────────────────────────────────────────────┤
│ 1  │ 指标A    │ 98.5  │ 2026-03-28 10:00  │
│ 2  │ 指标B    │ 76.2  │ 2026-03-28 10:00  │
│ ...│          │       │                    │
└────────────────────────────────────────────┘
共 12,345 行（估算），显示前 100 行
```

**后端接口：**
- `GET /api/db-manager/table-data` — 新增
  - 参数：`datasource_id`, `table_name`, `limit`(默认100), `offset`(默认0), `order_by`, `order_dir`
  - 返回：`{ rows: [...], columns: [...], total_estimate: number }`

**安全控制：**
- 仅限 admin 角色
- `limit` 上限 1000，防止大表拖垮连接
- 不支持 WHERE 条件（避免 SQL 注入风险，查询场景由 AI 自然语言查询模块覆盖）

### 5.3 表属性编辑（新增）

**功能点：**

| 操作 | 支持数据库 | SQL 示例 |
|---|---|---|
| 修改表注释 | MySQL, PostgreSQL, Oracle, 达梦, 人大金仓 | `ALTER TABLE t COMMENT='新注释'` / `COMMENT ON TABLE t IS '新注释'` |
| 表重命名 | 全部 7 种 | `ALTER TABLE old RENAME TO new` / `RENAME TABLE old TO new` |

**交互流程：**
- 在表名旁显示编辑图标
- 点击后弹出编辑框，修改表名或注释
- 先 SQL 预览，确认后执行
- 表重命名后自动刷新表列表并选中新表名

**后端接口：**
- `POST /api/db-manager/rename-table` — 新增
  - 参数：`datasource_id`, `old_name`, `new_name`, `execute`
- `POST /api/db-manager/update-table-comment` — 新增
  - 参数：`datasource_id`, `table_name`, `comment`, `execute`

### 5.4 表结构导出（新增）

**用户价值：**
导出建表 SQL，用于跨环境部署或文档归档。

**交互流程：**
1. 选中表 → 点击"导出 DDL"按钮
2. 弹出模态框显示完整的 CREATE TABLE SQL
3. 支持：复制到剪贴板 / 下载为 .sql 文件
4. 可选包含索引定义

**后端接口：**
- `GET /api/db-manager/export-ddl`
  - 参数：`datasource_id`, `table_name`, `include_indexes`(默认true)
  - 返回：`{ ddl: "CREATE TABLE ..." }`

**实现策略：**
- MySQL：`SHOW CREATE TABLE`
- PostgreSQL：通过 `information_schema` 拼装（或 `pg_dump --schema-only` 如可用）
- 其他数据库：基于已有的表结构查询结果，在后端拼装 CREATE TABLE SQL

### 5.5 Oracle/达梦/人大金仓 DDL 补齐

**现状：**
现有 DDL 操作（建表、加字段、改字段等）的 SQL 生成主要覆盖了 MySQL、PostgreSQL、SQLite。需要补齐 Oracle、达梦、人大金仓的方言差异。

**关键差异点：**

| 操作 | Oracle/达梦/人大金仓 差异 |
|---|---|
| 标识符引用 | 使用双引号 `"name"`（与 PostgreSQL 相同） |
| 自增主键 | Oracle 12c+ 用 `GENERATED ALWAYS AS IDENTITY`；达梦/金仓类似 |
| 字段注释 | 均用 `COMMENT ON COLUMN` 语法 |
| 表注释 | 均用 `COMMENT ON TABLE` 语法 |
| VARCHAR 类型 | Oracle 用 `VARCHAR2`，达梦/金仓兼容 `VARCHAR` |
| 布尔类型 | Oracle 无原生布尔，用 `NUMBER(1)`；达梦/金仓支持 `BOOLEAN` |

**实现方案：**
- 在现有 `_quote_ident` 基础上，扩展各数据库的 SQL 方言处理
- 新增 `_generate_column_def(col, db_type)` 统一函数，处理类型映射
- 对 `list_tables`、`get_table_structure` 补齐 Oracle/达梦/金仓的查询语句

## 6. 前端交互设计

### 6.1 页面布局重构

当前是"数据源选择 + 表选择 + 详情"的纵向堆叠布局。建议改为**左右分栏**：

```
┌──────────────────────────────────────────────────────────┐
│ 库表管理                                    [数据源: ▼]  │
├─────────────┬────────────────────────────────────────────┤
│ 表列表       │  result_daily — 日报结果表                │
│ ┌─────────┐ │                                            │
│ │🔍搜索... │ │  [表结构] [索引] [数据预览]               │
│ └─────────┘ │                                            │
│ 共 127 张    │  ┌──────────────────────────────────┐     │
│              │  │ 字段名 │ 类型 │ PK │ 可空 │ 注释 │     │
│ ▸ result_*   │  ├──────────────────────────────────┤     │
│   daily   ◀─│  │ id     │ INT  │ ✓  │ NO   │ 主键 │     │
│   monthly   │  │ name   │ VAR..│    │ YES  │ 名称 │     │
│   yearly    │  │ ...    │      │    │      │      │     │
│              │  └──────────────────────────────────┘     │
│ ▸ config_*   │                                            │
│   ...        │  [添加字段] [导出DDL]                      │
│              │                                            │
├─────────────┤                                            │
│ [+ 建表]    │                                            │
│ [↻ 刷新]    │                                            │
└─────────────┴────────────────────────────────────────────┘
```

**要点：**
- 左侧面板宽度约 240px，可折叠
- 表列表支持按前缀分组（如 `result_*`），可选功能
- 选中表高亮，右侧展示详情
- 建表、刷新按钮放在左侧面板底部

### 6.2 Tab 设计

选中表后，右侧详情区分为 3 个 Tab：

| Tab | 内容 | 操作 |
|---|---|---|
| 表结构 | 字段列表表格 | 添加字段、修改字段、删除字段 |
| 索引 | 索引列表表格 | 创建索引、删除索引 |
| 数据预览 | 前 100 行数据 | 只读浏览、排序 |

### 6.3 DDL 安全交互

所有 DDL 操作遵循统一的安全交互模式：

```
用户操作 → 生成 SQL 预览 → 用户确认 → 执行 → 结果反馈
           ↑                    ↑
        可复制/修改          二次确认弹窗
        （只读展示）         （高危操作）
```

**高危操作定义：**
- 删表（DROP TABLE）→ 强制二次确认 + 自动备份
- 删字段（DROP COLUMN）→ 二次确认
- 修改字段类型（可能丢数据）→ 警告提示 + 确认
- 表重命名 → 确认（可能影响依赖）

**低风险操作：**
- 建表、添加字段、创建索引 → SQL 预览后直接执行（无二次确认）

## 7. 权限控制

### 7.1 角色权限矩阵

| 操作 | admin | operator | viewer |
|---|---|---|---|
| 浏览表列表 | ✓ | ✓ | ✓ |
| 查看表结构 | ✓ | ✓ | ✓ |
| 数据预览 | ✓ | ✓ | ✗ |
| 建表 | ✓ | ✗ | ✗ |
| 修改表结构 | ✓ | ✗ | ✗ |
| 删表 | ✓ | ✗ | ✗ |
| 索引管理 | ✓ | ✗ | ✗ |
| 导出 DDL | ✓ | ✓ | ✗ |

**说明：**
- 当前所有 DDL 操作已限制为 `admin` 角色，保持不变
- 表浏览和结构查看可降低为 `operator` 角色，方便运维人员查看但不操作
- 数据预览需要 `operator` 以上角色（数据敏感性考虑）

### 7.2 审计日志

所有写操作（建表、改表、删表、索引变更）已接入审计日志（`log_operation`），保持现有机制不变。新增操作同样接入。

## 8. 与现有功能的关系

### 8.1 与纳管表配置的联动

```
库表管理（建表/改结构）  →  纳管表配置（选表/配置字段语义）  →  数据回写流程
         ↑                              ↓
    结构层准备                      业务层使用
```

**联动点：**
- 建表完成后，提供"立即纳管"快捷入口，跳转到纳管表配置页面并预选数据源和表名
- 添加字段后，提示"纳管表字段配置可能需要更新"

### 8.2 与 AI 自然语言查询的关系

- 库表管理：结构层操作（DDL）
- AI 自然语言查询：数据层操作（DQL/DML）
- 两者互补，不重叠

### 8.3 与数据源管理的关系

- 库表管理依赖数据源管理提供的连接信息
- 数据源的连接测试状态影响库表管理的可用性
- 库表管理不管理数据源本身

## 9. 数据同步功能评估

### 9.1 需求分析

用户提到的"数据同步"可能包含两类需求：

| 类型 | 场景 | 复杂度 | 建议 |
|---|---|---|---|
| 一次性表迁移 | 把 A 库的表（结构+数据）复制到 B 库 | 中 | P3 远期考虑 |
| 定时增量同步 | 源库数据变更后自动同步到目标库 | 高 | 暂不纳入 |

### 9.2 建议

**本期不做数据同步**，原因：
1. 产品定位是"轻量运维工作台"，不是 ETL 平台
2. 定时同步需要后台任务调度、增量检测、冲突处理，复杂度远超当前架构
3. 一次性迁移可以通过"导出 DDL + 数据导出/导入"组合实现

**远期路线：**
- P3 阶段评估一次性表迁移（结构克隆 + 数据插入）
- 定时同步建议引导用户使用专业 ETL 工具（如 DataX、SeaTunnel）

## 13. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| Oracle/达梦/人大金仓缺乏测试环境 | DDL 操作无法验证 | 优先用 MySQL/PostgreSQL 验证逻辑，达梦/金仓待环境就绪后补测 |
| 大表数据预览性能 | 超时或 OOM | 强制 limit 上限 1000，使用 `SELECT *` 而非聚合 |
| SQLite ALTER TABLE 限制 | 部分操作不可用 | 前端对 SQLite 禁用不支持的操作，给出提示 |
| DDL 操作误执行 | 数据丢失 | SQL 预览 + 二次确认 + 自动备份三重保护 |
