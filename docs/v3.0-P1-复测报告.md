## 一、多表变更对比复测（原 BUG #1）

### 原问题
> **BUG #1 — 多表/全局批量修改 500 内部错误 🔴**
> `build_explanation()` 中 `action['new_value']` → `KeyError`，导致多表模式和全局模式完全不可用。

### 复测结果

| # | 测试点 | 原结果 | 复测结果 | 说明 |
|---|--------|--------|----------|------|
| 3.10 | 多表模式 + 不匹配字段的表 | ❌ 500 | ✅ 通过 | cctv_polling 无匹配字段时返回 `status=error` + 提示"规则不适用于此表"，不再 500 |
| 3.11 | 全局模式遍历所有表 | ❌ 500 | ✅ 通过 | 遍历 18 张已纳管表，仅 zbap 有变更（30条），其余表优雅降级为 error 状态 |

### 详细复测数据

**多表模式（zbap + cctv_polling）：**
```
POST /api/ai/batch-fill-multi
  table_ids: [3, 16], scope: "multi"
  rule_text: "把所有值班时间中的：替换为:"

✅ HTTP 200
  session_id: 6e0b88d62c78
  summary: tables_affected=1, total_rows_changed=30, total_tables=2
  zbap: status=has_changes, rows_changed=30, explanation="将【值班时间】中的'：'替换为':'"
  cctv_polling: status=error, error="规则不适用于此表（替换操作缺少必要参数）"
```

**全局模式：**
```
POST /api/ai/batch-fill-multi
  table_ids: [3, 16], scope: "global"

✅ HTTP 200
  session_id: 52265bf623e0
  summary: tables_affected=1, total_rows_changed=30, total_tables=18
  zbap: has_changes（30条）
  其余 17 张表: error（规则不适用），无崩溃
```

**multi-confirm 确认接口：**
```
POST /api/writeback/multi-confirm
  session_id: "6e0b88d62c78"
  confirmations: [{"table_id": 3, "confirmed": true}]

✅ HTTP 200
  tables_success=1, tables_failed=0
  zbap: updated=30, backup_table=zbap_bak_20260324_142027_8801
```

**multi-confirm 异常场景：**
- ✅ 无效 session → 正确返回"多表预览数据不存在或已过期"
- ✅ session 已消费后重复使用 → 正确返回过期提示
