## 二、启动 Loading 页（6 项）

### ✅ 2.1 /api/startup-progress 接口返回阶段和百分比

**结论：通过**

- 后端 `startup_state.py` 定义了 4 个阶段：`initializing_db`（0-20%）、`loading_plugins`（20-50%）、`starting_server`（50-80%）、`ready`（100%）
- `main.py` 中 `get_startup_progress()` 路由注册在 lifespan 之外，确保启动过程中也能响应
- `to_dict()` 返回 `stage`、`stage_label`、`percent`、`ready` 四个字段，结构完整

### ✅ 2.2 前端 /loading 页面可正常加载

**结论：通过**

- `Loading.tsx` 存在且结构完整
- `App.tsx` 中注册了 `<Route path="/loading" element={<Loading />} />`
- 页面使用 Ant Design `Progress` 组件，有渐变配色和居中布局

### ✅ 2.3 进度条和阶段文字正确显示

**结论：通过**

- `Loading.tsx` 轮询 `/api/startup-progress`（每 1 秒），将 `percent` 传给 `<Progress>`，`stage_label` 作为文字显示
- 后端 `STAGE_LABELS` 提供中文阶段名：「正在初始化数据库...」「正在加载插件...」「正在启动服务...」「启动完成」
- 进度条 `status` 根据 `ready` 切换为 `'success'` 状态

### ✅ 2.4 进度 100% 后自动跳转登录页

**结论：通过**

- `Loading.tsx` 在 `res.data.ready === true` 时清除定时器，延迟 500ms 后 `navigate('/login', { replace: true })`
- 跳转使用 `replace: true`，不会在浏览器历史栈留下 /loading 记录

### ✅ 2.5 服务已就绪时直接访问 /loading 应立即跳转

**结论：通过**

- 首次加载即执行 `poll()`，如果后端已 ready（percent=100），第一次轮询就会触发跳转
- 500ms 延迟可接受

### 🐛 2.6 start.sh / start.bat / launcher.py 中 URL 已改为 /loading

**结论：有缺陷**

| 文件 | URL 变量 | HEALTH_URL 变量 | 结果 |
|------|----------|-----------------|------|
| `start.sh` | `http://localhost:8580/loading` ✅ | `${URL}/api/health` → `http://localhost:8580/loading/api/health` ❌ | **健康检查 URL 错误** |
| `start.bat` | `http://localhost:8580/loading` ✅ | `%URL%/api/health` → `http://localhost:8580/loading/api/health` ❌ | **健康检查 URL 错误** |
| `dist/start.sh` | 同上 ✅ | 同上 ❌ | **健康检查 URL 错误** |
| `dist/start.bat` | 同上 ✅ | 同上 ❌ | **健康检查 URL 错误** |
| `launcher.py` | `http://localhost:8580`（无 /loading） | 健康检查用 `{URL}/api/health` ✅，浏览器打开 `{URL}/loading` ✅ | **正确** |

**缺陷详情：**

> **严重级别：高**
>
> `start.sh` 和 `start.bat` 将 `URL` 改为 `http://localhost:8580/loading` 后，`HEALTH_URL` 从 `${URL}/api/health` 拼接，得到错误路径 `http://localhost:8580/loading/api/health`。
>
> **影响：** 健康检查不再命中 `/api/health` 端点。实际会命中 SPA catch-all（返回 index.html + 200），导致健康检查在 SPA catch-all 注册后才"碰巧"通过，时间窗口与预期不同。当 catch-all 未注册时（启动早期），返回 404，curl `-sf` 会失败，轮询仍可正常等待，但逻辑上是错误的。
>
> **修复建议：** 将 `HEALTH_URL` 改为独立拼接：
> ```bash
> # start.sh
> HEALTH_URL="http://localhost:${PORT}/api/health"
> ```
> ```bat
> :: start.bat
> set "HEALTH_URL=http://localhost:%PORT%/api/health"
> ```
> 4 个文件均需修复（start.sh、start.bat、dist 下两份）。

## 三、离线状态提示（6 项）

### ✅ 3.1 /api/system/network-status 接口返回 online/offline 状态

**结论：通过**

- `network_status.py` 注册 `GET /api/system/network-status`
- 返回 `{ "online": bool, "checked_at": str }`
- 探测逻辑尝试 4 个 DNS（Google、Cloudflare、AliDNS、114DNS），任一成功即 online
- 超时设置 3 秒合理

### ✅ 3.2 网络检测结果有缓存（60 秒内不重复检测）

**结论：通过**

- 后端 `_CACHE_TTL = 60`，`_cache["cached_until"] = now + 60`
- 60 秒内返回缓存结果，不重复 socket 探测
- 前端 `useNetworkStatus` 和 `aiGuard` 各有 30 秒客户端缓存（`CACHE_MS = 30_000`），进一步减少请求

### ✅ 3.3 AI 按钮在未配置 AI 模型时提示"请先配置"

**结论：通过**

- `aiGuard.ts` → `checkAIAvailable()` 第一步检查 `cfg.ai_enabled`
- 未启用时弹出 `Modal.warning`：标题「AI 功能未启用」，内容「请先在系统设置 → AI 配置中启用 AI 功能。」
- 语义符合"请先配置"要求

### ✅ 3.4 AI 按钮在离线时提示"需要联网"

**结论：通过**

- `aiGuard.ts` 第二步检查网络状态
- 离线时弹出 `Modal.warning`：标题「网络未连接」，内容「AI 功能需要联网使用，请检查网络连接。」
- 语义符合"需要联网"要求

### ✅ 3.5 AI 设置页顶部有网络状态指示器

**结论：通过**

- `AIConfigPage.tsx` 导入 `useNetworkStatus` hook（第 17 行）
