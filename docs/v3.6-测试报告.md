### ✅ 4. 最后一个管理员不可被禁用

**结论：通过**

后端实现了双重保护：

**禁用保护**（`update_user_status`）：
- 不可禁用自己（`target.id == user.id` 时拒绝）。
- 当目标用户角色为 admin 且操作为 disabled 时，查询启用状态的管理员数量，若 ≤ 1 则返回 400 拒绝。

**降权保护**（`update_user`）：
- 当目标用户当前角色为 admin 且新角色 ≠ admin 时，查询启用状态的管理员数量，若 ≤ 1 则返回 400「不能修改最后一个管理员的角色」。

两种路径均已覆盖，最后一个管理员无法被「禁用」或「降权」。

### ✅ 5. 禁用用户后该用户无法登录

**结论：通过**

- `get_current_user`（`backend/app/utils/auth.py`）查询用户时使用条件 `UserAccount.status == "enabled"`。
- 被禁用的用户（`status == "disabled"`）即使持有有效 JWT token，也无法通过认证。
- 登录接口（`auth.py` login）同样筛选 `status == "enabled"`。
- 已禁用用户尝试任何 API 调用 → 401。

### ✅ 11. 操作员/只读用户直接调用受限 API → 返回 403

**结论：通过**

`require_role` 装饰器实现（`backend/app/utils/auth.py`）：
```python
def require_role(*roles: str):
    def checker(user: UserAccount = Depends(get_current_user)):
        if user.role not in roles:
            raise HTTPException(status_code=403, detail=...)
        return user
    return checker
```

### ✅ 12. 操作日志中记录的是实际操作人（非硬编码 admin）

**结论：通过**

- `log_operation()` 函数签名中 `operator` 参数默认值为 `"admin"`，但所有实际调用处均显式传入实际操作人：
  - 用户管理路由：`operator=user.username`（user 来自 JWT 认证）
  - 数据维护路由：`operator=_get_username(user)`（`_get_username` 提取 `user.username`，无用户时返回 `"system"`）
- 审查所有 `log_operation` 调用点，未发现遗漏传 `operator` 参数的情况。
- 登录时记录 `last_login_at`（`auth.py`），确保操作人身份可追溯。

## 三、发现的缺陷

### 🟡 BUG-1：`get_import_diff` 和 `get_diff_report` 未限制角色（低风险）

**严重级别：P2（低）**

- `GET /import-tasks/{task_id}/diff`（行 1260）使用 `get_current_user`，未加 `require_role`。
- `GET /import-tasks/{task_id}/diff-report`（行 1296）同上。
- viewer 角色可查看差异预览和对比报告。

**分析**：这两个接口本质是「查看」操作（读取已有的导入任务差异），与 viewer 的只读定位不冲突。但如果严格理解为「导入相关功能」应限制操作员+，则需补充 `require_role("admin", "operator")`。

**建议处理方式**：确认产品意图 — 若 viewer 不应看到导入差异数据，则加角色限制；若允许查看，保持现状并在文档中说明。

### 🟡 BUG-2：`get_export_info` 未限制角色（低风险）

**严重级别：P2（低）**

- `GET /api/data-maintenance/{table_config_id}/export-info`（行 723）使用 `get_current_user`。
- viewer 角色可调用此接口获取导出模板信息。

**分析**：此接口返回导出相关元信息（字段映射等），viewer 前端虽然看不到导出按钮，但直接调用 API 可获取信息。实际数据安全影响低（不含敏感数据，且真正的导出接口已受保护）。

**建议处理方式**：建议补充 `require_role("admin", "operator")` 保持一致性，或标记为已接受风险。

### 🟢 提示：`log_operation` 默认参数 `operator="admin"` 存在误用风险

**严重级别：P3（提示）**

- 函数签名 `operator: str = "admin"` 意味着如果某个新增调用点忘记传 `operator`，日志将记录为 `"admin"` 而非实际操作人。
- **当前代码无此问题**，所有调用点均显式传参。
- 建议将默认值改为 `operator: str = "unknown"` 或移除默认值变为必传参数，防止未来遗漏。

**建议处理方式**：作为代码改进建议，不阻塞发布。
