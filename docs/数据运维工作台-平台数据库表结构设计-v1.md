# 数据运维工作台-平台数据库表结构设计-v1

## 1. 文档目的

本文档用于承接以下两份文档：

- 《数据运维工作台 PRD v1》
- 《数据运维工作台-页面原型结构与字段设计-v1》

目标是定义平台自身数据库的核心表结构，用于支撑以下能力：

- 数据源配置
- 纳管表配置
- 字段配置
- 模板导出记录
- 导入任务记录
- 回写日志
- 备份版本管理
- 系统操作日志

本文档优先满足 MVP 落地，不追求一次性大而全。

---

## 2. 设计原则

### 2.1 平台库与业务库分离
平台自身数据库只存：
- 配置
- 日志
- 任务
- 版本
- 元信息

业务数据库仍然存放被维护的结果表数据。

### 2.2 主键统一
建议平台自身所有表统一使用：
- `id`：bigint / snowflake / uuid 均可  
首版建议：`bigint`

### 2.3 审计字段统一
所有核心表建议统一保留以下审计字段：
- `created_by`
- `created_at`
- `updated_by`
- `updated_at`
- `is_deleted`

### 2.4 状态字段枚举化
尽量不要存自由文本状态，统一用枚举值或编码值。

### 2.5 不过度设计
首版先支撑单租户、单系统运行，不强行引入复杂租户模型。

---

## 3. 核心表清单

平台自身数据库首版建议至少包含以下 8 张核心表：

1. `datasource_config` 数据源配置表
2. `table_config` 纳管表配置表
3. `field_config` 字段配置表
4. `template_export_log` 模板导出日志表
5. `import_task_log` 导入任务日志表
6. `writeback_log` 回写日志表
7. `table_backup_version` 表备份版本记录表
8. `system_operation_log` 系统操作日志表

如需补充系统用户表，可后续增加，不纳入本次 MVP 核心范围。

## 4. 表结构设计

## 4.1 datasource_config（数据源配置表）

### 4.1.1 表用途
用于保存平台连接到外部业务数据库的数据源配置。

### 4.1.2 字段设计

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|
| id | bigint | 是 | 主键 |
| datasource_code | varchar(64) | 是 | 数据源编码，系统内唯一 |
| datasource_name | varchar(128) | 是 | 数据源名称 |
| db_type | varchar(32) | 是 | 数据库类型，如 mysql/postgresql/sqlserver |
| host | varchar(255) | 是 | 主机地址 |
| port | int | 是 | 端口 |
| database_name | varchar(128) | 否 | 默认数据库名 |
| schema_name | varchar(128) | 否 | 默认 schema |
| username | varchar(128) | 是 | 用户名 |
| password_encrypted | text | 是 | 加密后的密码 |
| charset | varchar(32) | 否 | 编码方式，默认 utf8 |
| connect_timeout_seconds | int | 否 | 连接超时秒数 |
| status | varchar(32) | 是 | 状态：enabled/disabled |
| last_test_status | varchar(32) | 否 | 最近测试结果：success/failed |
| last_test_message | varchar(1000) | 否 | 最近测试结果详情 |
| last_test_at | datetime | 否 | 最近测试时间 |
| remark | varchar(500) | 否 | 备注 |
| created_by | varchar(64) | 是 | 创建人 |
| created_at | datetime | 是 | 创建时间 |
| updated_by | varchar(64) | 是 | 更新人 |
| updated_at | datetime | 是 | 更新时间 |
| is_deleted | tinyint | 是 | 逻辑删除标记，0 否 1 是 |

### 4.1.3 约束与索引建议
- 主键：`id`
- 唯一索引：`uk_datasource_code(datasource_code)`
- 普通索引：
  - `idx_db_type(db_type)`
  - `idx_status(status)`

### 4.1.4 说明
- 密码必须加密存储，不可明文。
- 数据源编码建议系统自动生成，也可人工录入。

## 4.2 table_config（纳管表配置表）

### 4.2.1 表用途
用于登记哪些业务表被平台纳入维护，并保存对应的维护规则。

### 4.2.2 字段设计

| 字段名 | 类型 | 是否必填 | 说明 |
|---|---|---:|---|

| datasource_id | bigint | 是 | 关联数据源 ID |
| db_name | varchar(128) | 否 | 所属数据库名 |
| schema_name | varchar(128) | 否 | 所属 schema |
| table_name | varchar(128) | 是 | 原始表名 |
| table_alias | varchar(128) | 否 | 表中文名/别名 |
| table_comment | varchar(500) | 否 | 表说明 |
| config_version | int | 是 | 当前配置版本号 |
| structure_version_hash | varchar(128) | 否 | 表结构摘要 hash，用于快速比对 |
| primary_key_fields | varchar(500) | 是 | 主键字段列表，逗号分隔 |
| unique_key_fields | varchar(500) | 否 | 唯一识别字段列表，逗号分隔 |
| allow_export_current | tinyint | 是 | 是否允许导出当前数据 |
| allow_export_all | tinyint | 是 | 是否允许导出全量数据 |
| allow_import_writeback | tinyint | 是 | 是否允许上传回写 |
| allow_insert_rows | tinyint | 是 | 是否允许新增记录，首版默认 0 |
| allow_delete_rows | tinyint | 是 | 是否允许删除记录，首版默认 0 |
| backup_keep_count | int | 是 | 备份保留数量，默认 3 |
| strict_template_version | tinyint | 是 | 是否强校验模板版本 |
| strict_field_order | tinyint | 是 | 是否强校验字段顺序 |
| status | varchar(32) | 是 | 状态：enabled/disabled |
| structure_check_status | varchar(32) | 否 | 表结构状态：normal/changed/error |
| last_structure_check_at | datetime | 否 | 最近结构检查时间 |
| last_sync_at | datetime | 否 | 最近同步时间 |
| remark | varchar(500) | 否 | 备注 |
| created_by | varchar(64) | 是 | 创建人 |
| created_at | datetime | 是 | 创建时间 |
| updated_by | varchar(64) | 是 | 更新人 |
| updated_at | datetime | 是 | 更新时间 |
| is_deleted | tinyint | 是 | 逻辑删除 |

### 4.2.3 约束与索引建议
- 主键：`id`
- 唯一索引：
  - `uk_table_config_code(table_config_code)`
  - `uk_table_locator(datasource_id, db_name, schema_name, table_name, is_deleted)`
- 普通索引：
  - `idx_datasource_id(datasource_id)`
  - `idx_status(status)`
  - `idx_structure_check_status(structure_check_status)`

### 4.2.4 说明
- 这里的唯一标识不能只靠表名，必须结合数据源与库/schema。
- `structure_version_hash` 建议由字段名、字段类型、顺序等生成。

## 5. 表关系设计

## 5.1 关系总览

- 一个 `datasource_config` 可对应多个 `table_config`
- 一个 `table_config` 可对应多个 `field_config`
- 一个 `table_config` 可对应多个 `template_export_log`
- 一个 `table_config` 可对应多个 `import_task_log`
- 一个 `import_task_log` 可对应多个 `writeback_log`（通常首版一对一，但结构上不强绑）
- 一个 `table_config` 可对应多个 `table_backup_version`
- `system_operation_log` 为通用审计表，可关联任意模块对象

## 5.2 逻辑关系图（简化）

`datasource_config`
→ `table_config`
→ `field_config`

`table_config`
→ `template_export_log`
→ `import_task_log`
→ `writeback_log`

`table_config`
→ `table_backup_version`

## 10. MVP 表结构最小落地建议

若你希望第一版开发更快，可以先最小落地如下：

### 必做表
- `datasource_config`
- `table_config`
- `field_config`
- `import_task_log`
- `writeback_log`
- `table_backup_version`

### 建议一起做
- `template_export_log`
- `system_operation_log`

原因：
- 不做导出日志和系统日志，也能跑
- 但为了后续审计和演示完整度，建议首版一并纳入

## 11. 风险提醒

### 11.1 不建议把业务整表数据备份到平台库
整表备份建议仍保留在业务数据库中，平台库只记录备份元信息。

### 11.2 不建议首版就上复杂租户模型
会显著增加设计复杂度，但当前场景不需要。

### 11.3 不建议首版强行做逐字段审计
容易拖慢开发，首版做批次级日志就够。

### 11.4 不建议把字段映射完全依赖示例值
字段含义应由管理员确认后落表，不应只靠自动猜测。

## 12. 一句话总结

平台自身数据库的作用，不是替代业务库，而是为“数据源配置、表配置、字段规则、模板流转、导入校验、回写日志、备份版本、操作审计”提供一层稳定、可追踪、可管理的元数据与控制中枢。
