> 产出人：斯维因 | 时间：2026-03-26
> 状态：草案，待李琪确认

---

## 一、设计目标

把产品从"功能堆叠"升级为"成熟产品形态"：
1. 用户第一眼看到的是清晰的核心能力，不是 19 个插件平铺
2. 功能按价值分层，支持未来开源版 / 专业版 / 企业版的商业化拆分
3. 插件可按需启用，不用的不占导航空间
4. 为后续 InnerCat 功能封装、统一架构接入预留扩展点

## 二、三层架构

### 第一层：底座（Platform Core）
> 不可卸载，所有用户必备

| 功能模块 | 说明 | 导航位置 |
|---------|------|---------|
| 工作台总览 | 统计卡片、趋势图、待处理提醒、快捷入口 | 首页 |
| 数据源管理 | 数据源 CRUD、连接测试、7 种数据库支持 | 一级导航 |
| 纳管表配置 | 表配置、字段配置、结构检测 | 一级导航 |
| 数据浏览 | 分页查看、筛选、在线编辑、单行新增、标记删除 | 一级导航 |
| 模板导出/导入/回写 | 核心数据修订链路（含 INSERT/UPDATE/DELETE） | 一级导航（数据维护） |
| 版本回退 | 备份版本管理、一键回退 | 一级导航 |
| 日志中心 | 系统日志、导出/导入/回写日志 | 一级导航 |
| 用户与权限 | 三角色权限、用户管理、个人设置 | 一级导航 |
| 系统设置 | 基础配置 | 一级导航 |
| 登录认证 | JWT 认证（预留统一架构 OAuth 接入点） | - |

### 第二层：基础插件（Built-in Plugins）
> 默认启用，可由管理员关闭，覆盖日常运维高频场景

| 插件 | 功能 | 导航位置 |
|------|------|---------|
| 库表管理 `plugin_db_manager` | 查看/建表/改结构/删表/DDL | 一级导航"库表管理" |
| SQL 控制台 `plugin_sql_console` | 只读 SQL 查询 + 结果导出 | 一级导航"SQL 控制台" |
| 定时任务 `plugin_scheduler` | 定时巡检/备份/导出 | 一级导航"定时任务" |
| 健康巡检 `plugin_health_check` | 数据源连通性、表存在性、结构变化 | 一级导航"健康巡检" |
| 备份迁移 `plugin_backup` | 平台配置备份/导入/恢复 | 一级导航"备份迁移" |
| 批量操作 `plugin_batch_ops` | 批量纳管/批量导出/多表变更对比 | 一级导航"批量操作" |
| 通知中心 `plugin_notification` | 站内通知、未读管理 | 顶部铃铛图标 |

### 第三层：扩展插件（Extension Plugins）
> 默认关闭，用户在"插件中心"按需启用

| 分类 | 插件 | 功能 |
|------|------|------|
| **AI 智能** | `plugin_ai_assistant` | 字段推荐、自然语言查询、批量修改、智能校验、日志分析、影响评估 |
| **AI 智能** | `plugin_ai_predict` | 历史数据预测 + 预填模板 |
| **AI 智能** | `plugin_smart_import` | Word/PDF/Excel 智能解析 + 指标匹配 + 字段映射 |
| **数据治理** | `plugin_data_mask` | 字段级脱敏导出 |
| **数据治理** | `plugin_data_compare` | 跨库数据对比 |
| **数据治理** | `plugin_data_trend` | 行数变化趋势 + 预警 |
| **报表审计** | `plugin_audit_export` | PDF 审计报告 |
| **报表审计** | `plugin_report` | Excel/PDF 对比报告 |
| **流程协作** | `plugin_approval` | 审批流（回写需审批） |
| **外部集成** | `plugin_notify_push` | 企微/钉钉/邮件推送 |
| **外部集成** | `plugin_webhook` | Webhook 事件推送 |
| **资源市场** | `plugin_template_market` | 预设模板一键导入建表 |

## 三、插件中心页面设计

### 入口
- 一级导航底部新增「🧩 插件中心」入口
- 管理员可见，操作员和只读用户不可见（或只可查看不可操作）

### 页面结构
```
┌─────────────────────────────────────────────┐
│  🧩 插件中心                    [搜索框]     │
├─────────────────────────────────────────────┤
│  分类筛选：全部 | AI 智能 | 数据治理 |       │
│           报表审计 | 流程协作 | 外部集成      │
├─────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 🤖 AI    │  │ 📊 数据   │  │ 🔒 数据   │  │
│  │ 智能助手  │  │ 变更趋势  │  │ 脱敏导出  │  │
│  │          │  │          │  │          │  │
│  │ 7项AI能力 │  │ 趋势图+  │  │ 手机号/   │  │
│  │ 字段推荐  │  │ 预警     │  │ 身份证... │  │
│  │          │  │          │  │          │  │
│  │ [已启用✅]│  │ [启用]   │  │ [启用]   │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│  ...                                        │
└─────────────────────────────────────────────┘
```

### 插件卡片信息
- 图标 + 名称
- 一句话描述
- 分类标签
- 状态：已启用（绿）/ 未启用（灰）/ 需授权（锁）
- 操作：启用/停用开关（管理员）

### 启用/停用行为
- 启用：导航栏动态出现对应菜单入口
- 停用：导航栏移除，数据保留不删除
- 需授权：显示锁图标 + "联系管理员获取激活码"（预留统一架构接入）

## 五、对现有架构的改动评估

| 改动项 | 工作量 | 说明 |
|--------|--------|------|
| manifest.json 新增 `layer` 字段 | 小 | 值：`builtin` / `extension`，加载器据此分组 |
| manifest.json 新增 `category` 字段 | 小 | 值：`ai` / `governance` / `report` / `workflow` / `integration` / `market` |
| 插件中心前端页面 | 中 | 新页面，卡片列表 + 筛选 + 启用/停用 |
| 导航栏分组渲染 | 中 | 按 layer 分组，extension 层动态插入 |
| 插件启用状态持久化 | 小 | platform_db 新增 `plugin_status` 表 |
| 预留授权检查接口 | 小 | `GET /api/plugins/{id}/license-check`，当前默认返回 `allowed` |

**总评估：约 2-3 天开发量，不改底座逻辑。**

## 六、未来扩展预留

### InnerCat 功能封装
你提到的 InnerCat 功能可以作为扩展插件接入，比如：
- `plugin_innercat_chat`：内嵌对话窗口
- `plugin_innercat_agent`：AI Agent 自动化操作

manifest.json 中标记 `"source": "innercat"`，插件中心单独分组"InnerCat 能力"。

### 数据库定时备份（你提的新需求）
- 新增 `plugin_db_backup`：独立于 `plugin_backup`（平台配置备份）
- 支持：按表/按库、cron 表达式、保留最近 N 份、备份到本地/OSS
- 归入"基础插件"层

### 统一架构认证接入
- 预留 `GET /api/auth/oauth/callback` 端点
- 插件中心"需授权"状态对接统一架构的 license 服务

## 七、需要确认的决策点

1. **基础插件 7 个的划分是否合理？** 有没有要升到底座或降到扩展的？
2. **扩展插件默认全部关闭？** 还是部分默认开启（如 AI 智能助手）？
3. **数据库定时备份放基础插件还是扩展插件？**
4. **InnerCat 封装的优先级？** 是跟着这轮做还是后置？
5. **插件中心是否需要支持"在线安装新插件"？** 还是当前只管理已有插件的启用/停用？
