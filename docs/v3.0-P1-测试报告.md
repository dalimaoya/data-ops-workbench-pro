## 二、问题清单

### BUG #1 — 多表/全局批量修改 500 内部错误 🔴 严重

- **现象：** `POST /api/ai/batch-fill-multi` 当 `table_ids` 包含某些表（如 sp_sxtxx）或 `scope=global` 时，返回 HTTP 500 Internal Server Error
- **根因：** `batch_fill_engine.py:409` 行 `build_explanation()` 函数中，当 action type 为 `replace` 时，直接用 `action['old_value']` 和 `action['new_value']` 访问字典键。但 LLM 对某些表返回的 parsed_rule 中 action 可能缺少 `new_value` 键（例如 LLM 试图映射不存在的字段，返回了不完整结构）
- **堆栈：**
  ```
  ai_batch_fill_multi.py:286 → build_explanation(parsed_rule, fields_list)
  batch_fill_engine.py:409 → action['new_value'] → KeyError: 'new_value'
  ```
- **影响范围：** 多表模式和全局模式都受影响。单表模式不受影响（单表有独立的校验逻辑）
- **严重级别：** 🔴 高（多表和全局模式完全不可用）
- **建议修复：**
  1. `build_explanation()` 中使用 `action.get('old_value', '')` 和 `action.get('new_value', '')` 替代直接键访问
  2. `_process_single_table()` 中增加 parsed_rule 结构校验：如果 action type 为 replace 但缺少 old_value/new_value 键，应视为解析失败并返回 status=skipped
  3. 外层 `batch_fill_multi_preview()` 对 `_process_single_table` 加 try-except，单表报错不应阻断其他表处理

### 建议 #1 — 自然语言查询日期格式匹配 ⚠️ 低

- **现象：** 追问 "再加上日期是6月的" 时，LLM 返回 `rq LIKE "-06-"` 但 zbap 表中日期格式为 `2025.6.1`（点分隔非横线）
- **影响：** 追问结果可能返回空集
- **建议：** 在构建 field context 时加入 2-3 条数据样本，让 LLM 了解实际数据格式

### 建议 #2 — 规则引擎语义理解 ⚠️ 低

- **现象：** "把所有中文冒号替换为英文冒号" 在规则引擎（非 LLM）模式下，将"中文冒号"作为字面量搜索而非识别为 `：` 字符
- **影响：** 仅在 AI 功能未启用（降级到规则引擎）时出现
- **建议：** 在规则引擎中增加常见符号别名映射（中文冒号→：、英文冒号→:、中文句号→。等）
