## 二、逐项测试结果

### 测试点 1：标记待删除 → 导出 → 导入 → 校验识别删除标记

**结果：❌ 不通过（严重）**

**现象：**
- 导出模板正确：`export_template()` 函数已在 Excel 最后一列新增 `_操作` 列（第 627–643 行），表头有批注"填 DELETE 标记删除"，数据行单元格解锁可编辑，`_meta` 中写入 `"has_operation_column": True`。
- 导入校验缺陷：`import_template()` 函数（第 831–839 行）虽然检测到 `_操作` 列并记录了 `operation_col_idx`，但 **`operation_col_idx` 变量定义后从未被使用**。在第 6 步"读取行数据 & 校验"循环中，没有任何代码读取 `_操作` 列的值，也没有将带 DELETE 标记的行识别为待删除行。
- 因此，即使用户在 `_操作` 列填写了 `DELETE`，导入校验不会生成 `change_type: "delete"` 的差异记录。

**严重级别：🔴 P0 — 核心功能断裂**

**建议：** 在 `import_template()` 的行读取循环中，增加对 `_操作` 列值的读取逻辑，当值匹配 `_DELETE_MARKERS`（"DELETE"/"delete"/"删除"）时，将该行标记为待删除，并生成 `change_type: "delete"` 的 diff 记录。

---

### 测试点 2：差异预览正确区分三种操作类型的颜色

**结果：❌ 不通过（严重）**

**现象：**
- 前端 `DiffPreview.tsx` 的 `change_type` 列渲染（第 126–136 行）只定义了两种颜色映射：
  ```tsx
  const map = {
    update: { color: 'orange', label: '更新' },
    insert: { color: 'green', label: '新增' },
  };
  ```
  **缺少 `delete` 类型的映射**（应为红色）。若后端传入 `change_type: "delete"`，前端会显示 fallback 的默认灰色标签，而非需求中的红色。
- 后端差异对比报告 `get_diff_report()` 和 `generate_compare_report()` 中已正确定义三色（蓝/绿/红），但前端 DiffPreview 页面遗漏。
- `rowClassName` 也仅处理了 `insert` 类型的行背景色，没有 `delete` 行的红色背景。

**严重级别：🟠 P1 — 用户无法在差异预览中识别待删除行**

**建议：** 在 DiffPreview.tsx 的 change_type 映射中添加 `delete: { color: 'red', label: '删除' }`，并为 delete 行添加红色背景的 rowClassName。

---

### 测试点 3：DELETE 执行前自动备份

**结果：✅ 通过**

**分析：**
- `delete_rows()` 函数（第 2396 行起）在执行 DELETE 前，通过 `_create_backup_table()` 创建全表备份，逻辑与 UPDATE/INSERT 回写一致。
- 备份表命名格式：`{表名}_bak_{时间戳}_{随机后缀}`。
- 备份记录写入 `TableBackupVersion`，`trigger_type="triggered_by_delete"`。
- 支持多数据库方言（MySQL、PostgreSQL、SQL Server、Oracle、达梦、SQLite）。

---

### 测试点 4：DELETE 执行后数据确实从业务表删除

**结果：⚠️ 部分通过**

**分析：**
- 通过数据浏览页面的"删除选中行"功能（`DELETE /{table_config_id}/rows` 接口），可以正确执行 DELETE SQL，数据确实从业务表删除。
- 但通过**导入回写流程**（导出模板 → 标记 DELETE → 导入 → 差异预览 → 确认回写）无法触发 DELETE——因为导入校验未能识别删除标记（见测试点 1），回写函数 `writeback()` 中也**没有处理 `change_type: "delete"` 的逻辑**。
  - 回写函数（第 1660 行起）只处理了 `update_rows`（UPDATE）和 `new_rows_data`（INSERT），没有 DELETE 分支。
  - `diff_data.json` 中也没有存储 `delete_rows` 数据。

**严重级别：🔴 P0 — 模板回写 DELETE 路径完全不通**

---

### 测试点 5：结果页正确显示删除行数

**结果：❌ 不通过（严重）**

**现象：**
- 后端：`WritebackResult` 返回值（第 1780 行附近）中包含 `deleted_row_count=0`（固定为 0），无 `deleted` 字段返回给前端。回写结果 JSON 只返回 `updated` 和 `inserted` 字段。
- 前端：`WritebackResult` 接口定义中没有 `deleted` 字段。`DiffPreview.tsx` 的结果页展示（第 74 行）使用的 summary 模板 `writebackSummary` 只显示 `updated` 和 `inserted`，无删除数展示。
- 删除数统计卡片也缺失（差异预览只有"更新差异"和"新增行"卡片，无"删除行"卡片）。

**严重级别：🟠 P1 — 即使修复后，用户也看不到删除统计**

**建议：**
1. 后端 writeback 返回值增加 `deleted` 字段。
2. 前端 WritebackResult 接口增加 `deleted` 字段。
3. DiffPreview 结果页和差异统计卡片增加删除行数展示。

---

### 测试点 6：混合场景 — 同一批次含更新 + 新增 + 删除

**结果：❌ 不通过（阻塞于测试点 1）**

**分析：**
由于导入校验不识别 DELETE 标记，无法在同一批次中产生三种类型的差异记录。更新和新增可以混合工作（v3.9 已实现），但删除无法参与。

**严重级别：🔴 P0**

---

### 测试点 7：边界 — 标记删除后取消标记

**结果：✅ 通过（模板层面）**

**分析：**
- 导出模板中 `_操作` 列的单元格是解锁的（`unlocked_cell`），用户可以自由填写和清除 DELETE 标记。
- 由于导入校验不读取 `_操作` 列的值，所以即使写了 DELETE 也不会被识别——反向也成立：取消标记不会导致问题。
- 本条测试的 Excel 交互层面通过，但端到端流程因测试点 1 阻塞。

---

### 测试点 8：边界 — 删除不存在的行（主键已被删除）

**结果：⚠️ 部分通过**

**分析：**
- 通过数据浏览页面直接删除：`delete_rows()` 函数中，对每个 pk_value 执行 `SELECT ... WHERE ...` 获取行数据后再执行 `DELETE ... WHERE ...`。如果主键不存在：
  - `SELECT` 返回 None（row_before 为 None）
  - `DELETE` SQL 不报错但影响 0 行
  - 计为"成功"（success_count += 1），但不记录 change_log
  - **不会报错提示"行不存在"**，这是一个轻微缺陷。
- 通过导入回写路径：因测试点 1 阻塞，无法测试。

**严重级别：🟡 P2 — 删除不存在的行不报错、不区分，可能误导用户**

**建议：** 检查 `DELETE` 影响行数（cursor.rowcount），如果为 0，标记为"跳过"而非"成功"。

---

### 测试点 9：权限 — 只读用户不可标记删除

**结果：✅ 通过**

**分析：**
- 后端接口层面：
  - `delete_rows()` 使用 `require_role("admin", "operator")`，readonly 用户调用会返回 403。
  - `export_template()` 使用 `require_role("admin", "operator")`，readonly 用户无法导出模板（也就无法编辑 `_操作` 列）。
  - `import_template()` 使用 `require_role("admin", "operator")`，readonly 用户无法上传模板。
  - 数据浏览返回 `allow_delete_rows` 时会检查用户角色（第 277 行、第 477 行），readonly 用户返回 `allow_delete_rows: 0`。
- 前端层面：
  - `DataBrowse.tsx` 中 `canOperate = user?.role === 'admin' || user?.role === 'operator'`，readonly 用户看不到删除按钮。
  - 行选择功能也受 `canOperate && allowDelete` 门控。

---

### 测试点 10：回退 — 删除后可通过版本回退恢复

**结果：✅ 通过**

**分析：**
- `delete_rows()` 在执行删除前创建全表备份，备份记录 `can_rollback=1`。
- `backup_version.py` 的 `rollback_version()` 函数（第 172 行起）实现了完整回退流程：
  1. 先备份当前数据（pre-rollback 备份）
  2. 清空原表（TRUNCATE 或 DELETE FROM）
  3. 从备份表恢复数据（INSERT INTO ... SELECT FROM）
  4. 记录回退日志
- 回退操作限 admin 角色。
- 支持多数据库方言。

## 五、结论与建议

**v3.10 DELETE 回写的"导出标记 → 导入识别 → 差异预览 → 回写执行"完整链路存在断裂，核心逻辑未实现完成。**

具体来说：
1. **导出模板已正确添加 `_操作` 列**（✅ 完成）
2. **导入校验已检测到 `_操作` 列但未读取其值**（❌ 半成品）
3. **差异预览前端缺少 delete 类型展示**（❌ 未完成）
4. **回写执行缺少 DELETE 分支**（❌ 未完成）
5. **通过数据浏览页面直接删除行**的独立功能正常（✅ 已有，v2.0 实现）

**建议：**
- 需开发完成缺陷 #1、#2、#3、#4 后进行复测
- 建议优化缺陷 #5（P2），可排到下一迭代
- 建议开发提供修复后的 commit 号，巴德做回归验证
