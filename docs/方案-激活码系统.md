# 激活码系统方案

> 产出人：Claude Lead
> 日期：2026-03-29
> 状态：待用户确认

---

## 一、目标

1. 离线版用户可通过填入激活码解锁插件功能
2. 统一认证平台提供激活码生成管理页面
3. 激活码本地验证，不依赖网络

## 二、激活码生命周期

```
管理员在认证平台生成激活码
  → 通过公众号/邮件/线下发给用户
  → 用户在工作台填入激活码
  → 工作台本地验证激活码签名
  → 验证通过 → 写入本地 License
  → 插件解锁
```

## 三、激活码设计

### 3.1 格式

```
ACT:XXXX-XXXX-XXXX-XXXX
```

- 前缀 `ACT:` 固定
- 正文 16 位有效字符，分 4 组
- 排除易混淆字符：0/O、I/1、S/5

### 3.2 激活码内部结构（签名方案）

激活码本质是一个**签名 token**，包含：

```json
{
  "code": "ACT:K7M9-X4QP-T8LD-N2WR",
  "product": "data-ops-workbench",
  "plugin_keys": ["plugin_ai_assistant", "plugin_data_mask"],
  "expires_at": "2027-03-29T00:00:00Z",   // null = 永久
  "created_at": "2026-03-29T00:00:00Z",
  "batch_no": "BATCH-20260329-001",
  "signature": "base64_ed25519_signature"
}
```

### 3.3 本地验证方式

使用 **Ed25519 非对称签名**：
- 认证平台持有私钥，用于签名激活码
- 工作台内置公钥，用于验证签名
- 完全离线验证，不需要联网

验证流程：
1. 用户输入激活码字符串
2. 工作台用内置公钥验证签名
3. 检查 product 是否匹配
4. 检查 expires_at 是否过期
5. 检查激活码是否已被本地使用过（防重复）
6. 验证通过 → 写入本地 SQLite（activation_records 表）
7. 对应插件解锁

### 3.4 防伪造

- Ed25519 签名：没有私钥无法伪造激活码
- 公钥内置在程序中（编译到 Nuitka 二进制里），难以篡改
- 本地记录已使用的激活码，同一码不能重复激活

## 四、认证平台管理页面

### 4.1 功能

在现有管理后台（x-admin-key 鉴权）基础上，增加一个简单的 Web 管理页面：

| 功能 | 说明 |
|------|------|
| 生成激活码 | 选择产品、插件、有效期、数量 → 批量生成 |
| 查询激活码 | 按批次、状态、产品筛选 |
| 撤销激活码 | 标记为 revoked（离线场景下本地已激活的不受影响） |
| 导出激活码 | CSV 导出，用于线下分发 |

### 4.2 接口（已有基础）

认证底座已有：
- `POST /api/admin/activation-code` — 生成
- `GET /api/admin/activation-code` — 查询
- `POST /api/admin/activation-code/:code/revoke` — 撤销

需要新增：
- `POST /api/admin/activation-code/generate-offline` — 生成离线签名激活码（用 Ed25519 签名）
- `GET /api/admin/activation-code/export` — CSV 导出

### 4.3 管理页面

简单的单页面，放在认证底座的 `/admin` 路由下：
- 不做复杂前端框架，用原生 HTML + 简单 JS 即可
- x-admin-key 做鉴权
- 或者直接在现有的 Swagger UI 上操作也可以

## 五、工作台侧改动

### 5.1 激活码入口

在工作台的"系统设置"或"关于"页面增加：

```
┌─────────────────────────────┐
│  插件激活                    │
│                             │
│  激活码：[____________]     │
│                             │
│  [激活]                     │
│                             │
│  已激活插件：               │
│  ✅ AI 智能助手 (永久)      │
│  ✅ 数据脱敏导出 (至2027-03)│
└─────────────────────────────┘
```

### 5.2 数据模型

在 platform.db 新增表：

```sql
CREATE TABLE activation_record (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code VARCHAR(64) UNIQUE NOT NULL,       -- 激活码
  product VARCHAR(64) NOT NULL,            -- 产品标识
  plugin_keys TEXT NOT NULL,               -- JSON 数组
  expires_at DATETIME,                     -- null=永久
  activated_at DATETIME NOT NULL,          -- 激活时间
  signature TEXT NOT NULL                  -- 原始签名（用于验证）
);
```

### 5.3 内置公钥

Ed25519 公钥文件放在 `data/activation_public.pem`，Nuitka 编译时打包进去。

## 六、安全考量

1. **离线签名不可撤销** — 已激活的离线码，即使认证平台标记 revoked，本地也不受影响（离线场景的固有限制）
2. **时钟篡改** — 用户修改本地时间可以绕过过期检查。可接受风险（政企内网场景时钟通常由域控管理）
3. **公钥替换** — Nuitka 编译后公钥在二进制里，普通用户无法替换。高级用户仍可能反编译，但成本远高于购买激活码
4. **激活码泄露** — 一码一用（本地记录），但不同实例可以用同一码。如需限制，未来可加设备绑定
