## 一、功能定位

### 1.1 是什么

智能数据导入中心是数据运维工作台的新增核心模块，专门处理**非标数据源的智能解析与入库**。它是现有"平台模板导入"的高级补充——平台模板处理标准化流程，智能导入中心处理来自外部的非标准数据。

### 1.2 解决什么问题

在政企数据运维场景中，数据来源远不只是"下载平台模板 → 本地修改 → 上传回来"这一种。真实情况：

| 数据来源 | 典型场景 | 当前处理方式 |
|---|---|---|
| 统计局下发的 Word/PDF 报表 | 年度统计报告中的表格数据 | 人工抄到 Excel，再粘贴到数据库 |
| 客户整理的指标表格 | 列名是业务语言，和数据库字段名不对应 | 人工逐列对照映射 |
| 月度数据 → 年度库 | 客户给的是月维度，数据库存年维度 | 人工做汇总计算 |
| 一个 Sheet 里混了多个指标 | 多组数据混排，格式不规范 | 人工拆分后逐个处理 |
| 多表数据一次性提交 | 一份文件包含多张指标表的数据 | 人工识别分类后逐表录入 |

**核心痛点：** 数据进来的格式千差万别，但最终都要准确地更新到对应的数据库表和字段中。中间的"格式转换 + 字段对照 + 维度统计"全靠人工，效率低、易出错。

### 1.3 不做什么

- 不替代现有的平台模板导入流程（标准流程仍然推荐走模板）
- 不做 ETL 平台（不处理数据清洗、转换的复杂 Pipeline）
- 不做自由格式 Excel 的无条件接入（必须有 AI 识别 + 人工确认环节）
- 不做实时数据对接（只处理文件级的批量导入）

## 二、支持的输入格式

| 格式 | 文件类型 | 解析方式 | 备注 |
|---|---|---|---|
| Excel | .xlsx / .xls | openpyxl / xlrd 直接读取 | 主要格式 |
| Word | .docx | python-docx 提取表格 | 提取文档中的所有表格 |
| PDF | .pdf | pdfplumber + camelot | 结构化表格优先 pdfplumber，复杂排版用 camelot |
| CSV | .csv | pandas 直接读取 | 简单场景 |

## 三、五大核心场景设计

### 场景 1：Word/PDF 统计报表解析

#### 3.1.1 场景描述

用户上传来自统计局、上级单位或客户的 Word/PDF 统计报表，系统自动提取其中的表格数据，识别对应的纳管表和字段，完成入库。

#### 3.1.2 处理流程

```
上传 Word/PDF 文件
  │
  ├─ 文件解析层（不依赖 AI）
  │   ├─ Word: python-docx 遍历 document.tables，提取每个表格
  │   ├─ PDF: pdfplumber.extract_tables()，逐页提取表格
  │   └─ 输出：多个二维数组（每个表格一个）
  │
  ├─ 表格清洗（规则引擎）
  │   ├─ 去除空行空列
  │   ├─ 识别表头行（第一非空行 / 加粗行）
  │   ├─ 合并单元格展开
  │   ├─ 数值格式统一（去除逗号、百分号转小数等）
  │   └─ 输出：清洗后的结构化表格数据
  │
  ├─ 进入通用的「指标表匹配 → 字段映射 → 预览回写」流程
  │
  └─ 特殊处理：
      ├─ PDF 图片表格：提示"检测到图片格式表格，无法自动提取，建议手动输入"
      └─ 多页跨页表格：自动拼接同结构的连续页表格
```

### 场景 2：客户指标表的智能字段映射

#### 3.2.1 场景描述

客户提交的 Excel 指标表，列名使用业务语言（如"全市地区生产总值(亿元)"），需要映射到数据库字段（如 `gdp_total`）。有时客户自己标注了对应字段名，有时没有。

#### 3.2.2 映射策略（三层匹配）

```
客户列名匹配目标字段
  │
  ├─ 第一层：精确匹配（规则引擎）
  │   客户列名 == 字段名 or 字段别名（语义名）
  │   命中 → 置信度 1.0，直接映射
  │
  ├─ 第二层：模糊匹配（规则引擎）
  │   ├─ 去除括号内容后匹配（"全市GDP(亿元)" → "全市GDP"）
  │   ├─ 同义词表匹配（"生产总值" ↔ "GDP"）
  │   ├─ 包含关系（客户列名包含字段语义名，或反过来）
  │   └─ 命中 → 置信度 0.6-0.9，展示推荐
  │
  ├─ 第三层：AI 语义匹配
  │   把客户列名和所有目标字段语义名发给 AI
  │   AI 返回每个客户列名最可能对应的字段
  │   命中 → 置信度取 AI 返回的分数
  │
  └─ 未命中 → 置信度 0，标记为"未匹配"，用户手动指定
```

### 场景 3：时间维度不一致的自动转换

#### 3.3.1 场景描述

数据库中存储的是年度汇总数据，客户提供的是月度/季度明细数据，需要系统识别维度差异并做聚合转换。

#### 3.3.2 维度检测逻辑（规则引擎）

```
Step 1: 检测源数据的时间维度
  ├─ 列头/行头包含 "1月-12月" / "Jan-Dec" → 月度
  ├─ 列头/行头包含 "Q1-Q4" / "一季度-四季度" → 季度
  ├─ 列头/行头包含 "上半年/下半年" → 半年度
  ├─ 列头/行头包含 "2023/2024/2025" → 年度
  └─ 无法识别 → 提示用户手动指定

Step 2: 获取目标表的时间维度
  ├─ 从字段配置中获取时间字段的粒度标记
  └─ 或从现有数据中推断（查看时间字段的值分布）

Step 3: 比较维度
  ├─ 源 == 目标 → 不需要转换
  ├─ 源更细（月→年） → 需要聚合
  ├─ 源更粗（年→月） → 提示"无法拆分，建议直接按年度导入"
  └─ 维度不可比 → 提示用户处理
```

#### 3.3.3 聚合方式配置

在字段配置中新增「聚合方式」属性：

| 聚合方式 | 适用场景 | 示例 |
|---|---|---|
| 求和 (SUM) | 累加型指标 | GDP、财政收入、投资额 |
| 平均值 (AVG) | 均值型指标 | 平均工资、平均气温 |
| 加权平均 | 比率型指标 | 增长率、占比 |
| 取最新 (LAST) | 时点型指标 | 人口总数、期末余额 |
| 取最大 (MAX) | 峰值型指标 | 最高气温、峰值产量 |
| 取最小 (MIN) | 谷值型指标 | 最低气温 |

> 如果字段未配置聚合方式，系统默认使用「求和」，并标黄提醒用户确认。

### 场景 4：混合 Sheet 的智能区域分割

#### 3.4.1 场景描述

客户的 Excel 不规范，一个 Sheet 里放了多组不同指标的数据。每组的起始行、列数、表头都不同，中间可能用空行或标题行分隔。

#### 3.4.2 区域分割策略

**基础模式（规则引擎，不依赖 AI）：**

```
扫描 Sheet 的每一行
  │
  ├─ 检测分隔特征：
  │   ├─ 连续空行（≥ 1 行全空）→ 区域分界
  │   ├─ 合并单元格行 → 可能是标题行（区域起始）
  │   ├─ 格式突变行（加粗/字号变大/背景色变化）→ 可能是标题行
  │   ├─ 列数突变（前面 5 列有数据，突然变成 8 列）→ 可能是新区域
  │   └─ 数据类型突变（从数值变成文本）→ 可能是新区域的表头
  │
  ├─ 基于分界标记，切分为多个区域
  │   每个区域包含：起始行、结束行、起始列、结束列、表头行
  │
  └─ 输出分割结果 + 每个区域的预览
```

**增强模式（AI 辅助，需要大模型）：**

```
将 Sheet 内容序列化为文本（保留行号和格式信息）
  │
  ├─ 发送给 AI，Prompt：
  │   "以下是一个 Excel Sheet 的内容，请识别其中有几组独立的数据表格，
  │    每组的起始行号、结束行号、列范围、表头行号是什么。
  │    返回 JSON 格式。"
  │
  └─ AI 返回分割建议
```

**兜底模式（用户手动）：**

```
提供可视化预览界面
  │
  ├─ 显示 Sheet 的全部内容（类似 Excel 预览）
  │
  ├─ 用户可以：
  │   ├─ 鼠标拖选一个矩形区域 → 标记为一组数据
  │   ├─ 指定某行为表头行
  │   └─ 添加多个区域
  │
  └─ 确认后继续处理
```

### 场景 5：多表数据自动识别与分发

#### 3.5.1 场景描述

用户一次性上传的文件中包含多张指标表的数据。系统需要自动识别每组数据对应的纳管表，并分发到正确的目标表。

#### 3.5.2 匹配策略

```
对每个已分割的数据区域：
  │
  ├─ 第一层：标题匹配
  │   区域标题 vs 纳管表别名
  │   "人口统计" → 人口统计表 (population_stats)
  │
  ├─ 第二层：列名匹配
  │   区域的列名集合 vs 各纳管表的字段语义名集合
  │   计算 Jaccard 相似度，取最高分的表
  │
  ├─ 第三层：数据特征匹配
  │   数据值范围 vs 各纳管表历史数据的值范围
  │   如：某列值在 0-2000 之间，历史上只有 GDP 表有这个量级
  │
  ├─ 第四层：AI 语义匹配（可选）
  │   把区域的列名+样本数据发给 AI
  │   AI 推荐最可能的目标表
  │
  └─ 综合评分，按置信度排序，展示给用户
```

#### 3.5.3 页面设计

| 条件 | 跳过步骤 | 说明 |

## 六、AI 依赖度分析与降级策略

| 步骤 | 规则引擎 | 统计方法 | AI（小模型） | AI（大模型） | 完全无 AI 可用？ |
|---|---|---|---|---|---|
| 文件解析 | ✅ 主力 | — | — | — | ✅ 完全可用 |
| 区域分割 | ✅ 基础（70%） | — | ⚠️ 增强 | ✅ 最佳 | ✅ 规则+手动 |
| 指标匹配 | ✅ 标题匹配 | ✅ 列名相似度 | ✅ 语义增强 | ✅ 最佳 | ✅ 规则+手动 |
| 字段映射 | ✅ 精确+模糊 | — | ✅ 语义增强 | ✅ 最佳 | ✅ 规则+手动 |
| 维度转换 | ✅ 完全可用 | — | — | — | ✅ 完全可用 |
| 数据校验 | ✅ 硬规则 | ✅ 异常检测 | ✅ 增强 | ✅ 最佳 | ✅ 完全可用 |

**结论：** 所有步骤在无 AI 情况下都可工作（规则引擎 + 用户手动），AI 是锦上添花。

## 七、数据流向安全保障

智能导入的数据最终走的是和平台模板相同的回写链路，所有安全机制完整保留：

| 安全点 | 说明 |
|---|---|
| 字段映射确认 | 用户必须确认每个字段的映射关系 |
| 维度转换确认 | 用户必须确认聚合方式和结果 |
| 数据校验 | 类型/必填/枚举/长度/AI异常值全套校验 |
| 差异预览 | 逐行逐字段展示原值→新值 |
| 多表逐表确认 | 涉及多表时逐表确认或批量确认 |
| 写前备份 | 每张表回写前自动备份 |
| 回写日志 | 记录逐字段变更明细 |
| 版本回退 | 出问题可一键回退 |
| 审批流 | 如开启，操作员的回写需管理员审批 |
