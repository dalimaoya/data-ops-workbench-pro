## 二、P0 阻断性 BUG

### BUG-001：SPA catch-all 路由拦截所有插件 GET API（P0 / 阻断）

**现象**：所有插件注册的 GET API 返回 `{"detail":"Not Found"}`（HTTP 200），但后端日志显示 200 OK。

**根因**：`main.py` 第 163 行的 SPA catch-all 路由 `@app.get("/{full_path:path}")` 在**模块级别**注册，早于插件路由的注册（插件在 `lifespan` startup 时才通过 `load_plugins()` 注册路由）。FastAPI 按注册顺序匹配，catch-all 中对 `api/` 前缀直接返回 `{"detail":"Not Found"}`。

**影响范围**：所有插件的 GET 路由，包括：
- `GET /api/notifications` — 通知列表
- `GET /api/approvals` — 审批列表
- `GET /api/settings/approval-enabled` — 审批开关
- `GET /api/health-check/config` — 巡检配置
- `GET /api/health-check/history` — 巡检历史
- `GET /api/scheduler/tasks` — 定时任务列表
- `GET /api/platform/backup/history` — 备份历史
- `GET /api/db-manager/tables` — 表列表
- `GET /api/db-manager/table-structure` — 表结构
- `GET /api/ai/config` — AI 配置（**此接口由核心路由注册，不受影响**）
- `GET /api/ai/import/mapping-templates` — 映射模板

**不受影响**：POST、PUT、DELETE 方法的插件路由（catch-all 只注册了 `@app.get`）。

**修复建议**：
1. 将 SPA catch-all 路由延迟注册到 `lifespan` startup 中，在 `load_plugins()` **之后**注册；
2. 或改用 Starlette 的 middleware 方式处理 SPA fallback。

## 四、其他发现

### 4.1 定时任务加载警告（P2 / 低优先级）

启动日志：`Failed to load task 1: 'Job' object has no attribute 'next_run_time'`

**影响**：已有定时任务无法自动恢复。不阻断功能使用，可手动重新创建。

### 4.2 健康巡检表名映射问题（P2 / 已知）

巡检显示 12 个"源表不存在"错误，但实际表（sp_sxtxx/zbap/cctv_polling）数据可正常浏览。巡检使用的是 `display_name` 而非 `table_name`，导致误报。非 v3.2 引入。

### 4.3 库表管理 SQL 生成不支持 AUTO_INCREMENT（P3 / 建议优化）

建表时 `is_primary_key: true` 的 INT 字段未自动添加 `AUTO_INCREMENT`。ColumnDef Schema 中也无此选项。建议增加 `auto_increment` 字段。

## 六、发布建议

### 🔴 不建议发布

**理由**：
1. P0 BUG（BUG-001）导致**所有插件的 GET 接口不可用**，前端 6 个插件页面完全不能使用
2. 库表管理新功能的表列表和表结构查看也被阻断
3. 核心功能虽然正常，但插件架构作为 v3.2 的核心变更，插件功能不可用等于 v3.2 主要目标未达成

**修复工作量预估**：BUG-001 约 10 分钟即可修复（调整路由注册顺序），修复后需重新验证。

**建议流程**：
1. 开发修复 BUG-001（调整 SPA catch-all 注册时序）
2. 巴德做 fix 后复测（仅验证插件 GET 接口恢复）
3. 通过后即可发布 v3.2
