## 二、安全加固需求清单

### 2.1 SQL 注入防护（P0 🔴）

**现状问题：** 34 处使用 f-string 拼接 SQL，主要集中在 `data_maintenance.py`、`backup_version.py`、`remote_db.py`。虽然表名和列名来自系统配置（非用户直接输入），但仍需加固。

**加固方案：**

| 措施 | 说明 |
|---|---|
| 表名/列名白名单校验 | 所有动态表名、列名必须与数据库实际 schema 比对，不在白名单内的拒绝执行 |
| 标识符转义函数 | 统一使用安全的标识符引用函数（如 `quote_identifier()`），禁止裸拼接 |
| 数据值全部参数化 | 确保所有 WHERE 条件、INSERT/UPDATE 的值都使用参数化占位符 |
| SQL 注入检测中间件 | 对用户输入的搜索关键字、筛选值做 SQL 关键词检测（SELECT/DROP/UNION 等） |

**验收标准：**
- [ ] 所有 f-string SQL 中的表名/列名经过白名单校验
- [ ] 所有用户输入值使用参数化查询
- [ ] 搜索框和筛选条件输入做 SQL 关键词过滤
- [ ] 单元测试覆盖常见注入向量（' OR 1=1、UNION SELECT、'; DROP TABLE 等）

### 2.2 密码存储升级（P0 🔴）

**现状问题：** 使用 SHA256 + 固定盐值（`_SALT`），不抗彩虹表和暴力破解。

**加固方案：**
- 替换为 **bcrypt**（Python `passlib` 或 `bcrypt` 库）
- 每个密码独立随机盐
- 数据库迁移：首次升级时，旧密码在用户下次登录时自动转为 bcrypt 哈希

**验收标准：**
- [ ] 新创建的用户密码使用 bcrypt 存储
- [ ] 旧密码登录后自动迁移为 bcrypt
- [ ] 密码哈希长度 ≥ 60 字符（bcrypt 标准）

### 2.3 CORS 加固（P1 🟡）

**现状问题：** `allow_origins=["*"]` 允许所有来源跨域请求。

**加固方案：**
- 默认限制为同源（不设置额外 origins）
- 配置项 `ALLOWED_ORIGINS` 环境变量，管理员可指定允许的域名
- 开发模式下可放开

```python
origins = os.environ.get("ALLOWED_ORIGINS", "").split(",")
if not origins or origins == [""]:
    origins = []  # 同源策略
```

**验收标准：**
- [ ] 生产模式下 CORS 不允许 `*`
- [ ] 通过环境变量可配置允许的源
- [ ] 跨域请求被正确拦截

### 2.4 安全响应头（P1 🟡）

**需要添加的 HTTP 响应头：**

| 头 | 值 | 作用 |
|---|---|---|
| `X-Frame-Options` | `DENY` | 防止点击劫持 |
| `X-Content-Type-Options` | `nosniff` | 防止 MIME 类型嗅探 |
| `X-XSS-Protection` | `1; mode=block` | 浏览器 XSS 过滤 |
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline'` | 防止 XSS 注入外部脚本 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | 控制 Referrer 泄露 |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | 禁用非必要浏览器 API |

**实现方式：** FastAPI 中间件统一添加。

**验收标准：**
- [ ] 所有响应包含上述安全头
- [ ] 使用 [securityheaders.com](https://securityheaders.com) 或 curl 验证

### 2.5 API 限流（P1 🟡）

**加固方案：**

| 接口 | 限流规则 | 说明 |
|---|---|---|
| `/api/auth/login` | 5 次/分钟/IP | 防止暴力破解 |
| `/api/auth/captcha` | 10 次/分钟/IP | 防止验证码轰炸 |
| 数据导出 | 10 次/分钟/用户 | 防止大量导出消耗资源 |
| 回写操作 | 5 次/分钟/用户 | 防止频繁写入 |
| 其他 API | 60 次/分钟/用户 | 通用限流 |

**实现方式：** 使用 `slowapi` 库（基于 `limits`）或自定义中间件。

**验收标准：**
- [ ] 登录接口超频后返回 429 Too Many Requests
- [ ] 限流规则可通过配置调整
- [ ] 正常使用不受影响

### 2.6 登录安全加固（P1 🟡）

| 措施 | 说明 |
|---|---|
| 登录失败锁定 | 连续 5 次失败后锁定账号 15 分钟 |
| 登录失败日志 | 记录失败的用户名、IP、时间 |
| 密码强度要求 | 新密码至少 8 位，包含大小写+数字 |
| 会话管理 | 支持管理员强制下线某用户（废弃其 token） |

**验收标准：**
- [ ] 连续 5 次错误密码后账号锁定
- [ ] 锁定期间即使密码正确也无法登录
- [ ] 锁定信息在用户管理页面可见
- [ ] 管理员可手动解锁

### 2.7 文件上传安全（P1 🟡）

| 措施 | 说明 |
|---|---|
| 文件类型白名单 | 只允许 .xlsx / .xls 格式（后续加 .docx / .pdf / .csv） |
| 文件大小限制 | 单文件最大 50MB（可配置） |
| 文件名清洗 | 过滤路径穿越字符（`../`、`\`） |
| 上传目录隔离 | 上传文件存储在专用目录，不可被直接访问 |
| 病毒扫描 | 可选，集成 ClamAV（政企环境可能需要） |

**验收标准：**
- [ ] 非白名单文件类型上传返回 400
- [ ] 超大文件上传返回 413
- [ ] 路径穿越尝试被拦截

### 2.8 XSS 防护（P2 🟢）

| 措施 | 说明 |
|---|---|
| 后端输出过滤 | API 返回的用户数据中的 `<script>` 等标签做转义 |
| 前端渲染 | React 默认转义（已具备），注意 `dangerouslySetInnerHTML` 禁用 |
| CSP 头 | 见 2.4 安全响应头 |

### 2.9 JWT 密钥管理（P1 🟡）

**现状：** 默认密钥硬编码 `data-ops-workbench-secret-key-2026`。

**加固方案：**
- 首次启动时自动生成随机密钥并保存到配置文件
- 如果环境变量 `JWT_SECRET` 已设置，优先使用
- 启动时检测是否使用默认密钥，如果是则在日志中输出警告

**验收标准：**
- [ ] 首次启动自动生成随机密钥
- [ ] 使用默认密钥时控制台有明显警告
- [ ] 密钥存储在配置文件中，不在代码里

### 2.10 数据源密码加密存储（P1 🟡）

**现状：** 需确认数据源连接密码的存储方式。

**加固方案：**
- 数据源密码使用 AES 对称加密存储
- 加密密钥基于 JWT_SECRET 派生
- API 返回数据源信息时密码字段返回 `***`

**验收标准：**
- [ ] 数据库中存储的是加密后的密码
- [ ] API 不返回密码明文
- [ ] 编辑数据源时密码字段为空（需重新输入才更新）
