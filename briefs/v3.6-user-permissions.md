# v3.6 — 用户权限管理

## 版本目标
完善多用户使用场景，支持角色权限控制，确保不同用户只能做被授权的操作。

---

## 一、角色定义

| 角色 | 权限范围 |
|------|---------|
| 管理员（admin） | 全部功能 + 用户管理 + 系统设置 |
| 操作员（operator） | 数据浏览、模板导出导入、回写、日志查看 |
| 只读（viewer） | 仅数据浏览和日志查看，不可做任何写操作 |

初始安装时自动创建 admin 账号（现有逻辑保持）。

---

## 二、用户管理功能（仅管理员可用）

### 2.1 用户列表页
- 路径：系统设置 → 用户管理
- 展示：用户名、显示名、角色、状态（启用/禁用）、创建时间、最后登录时间
- 操作：新增、编辑、禁用/启用、重置密码

### 2.2 新增用户
- 字段：用户名（唯一）、显示名、角色（下拉选择）、初始密码
- 用户名规则：英文字母+数字+下划线，4-32 位
- 初始密码规则：至少 8 位

### 2.3 编辑用户
- 可修改：显示名、角色
- 不可修改：用户名

### 2.4 禁用/启用
- 禁用后该用户无法登录
- 不可禁用自己
- 不可禁用最后一个管理员

### 2.5 重置密码
- 管理员可重置其他用户的密码
- 重置后用户下次登录需要修改密码（可选，v3.6 不强制）

---

## 三、个人设置（所有用户可用）

### 3.1 修改密码
- 需验证旧密码
- 新密码至少 8 位
- 新密码不能与旧密码相同

### 3.2 修改显示名
- 用户可修改自己的显示名

### 3.3 个人信息展示
- 顶部导航栏显示当前登录用户名和角色
- 点击头像/用户名展开下拉菜单：个人设置、退出登录

---

## 四、权限控制

### 4.1 前端权限隔离

根据角色隐藏不可用的菜单和操作按钮：

| 功能菜单/操作 | admin | operator | viewer |
|-------------|-------|----------|--------|
| 仪表盘 | ✅ | ✅ | ✅ |
| 数据源管理 | ✅ | ❌ | ❌ |
| 纳管表配置 | ✅ | ❌ | ❌ |
| 字段配置 | ✅ | ❌ | ❌ |
| 数据浏览 | ✅ | ✅ | ✅ |
| 模板导出 | ✅ | ✅ | ❌ |
| 模板导入/回写 | ✅ | ✅ | ❌ |
| 日志查看 | ✅ | ✅ | ✅ |
| 备份管理 | ✅ | ❌ | ❌ |
| 系统设置 | ✅ | ❌ | ❌ |
| 用户管理 | ✅ | ❌ | ❌ |
| AI 功能 | ✅ | ✅ | ✅ |

### 4.2 后端权限校验

- 每个 API 路由添加角色校验装饰器/中间件
- 实现方式：在 auth 中间件解析 JWT token 后，提取用户角色
- 提供装饰器 `@require_role("admin")` `@require_role("operator", "admin")` 等
- 越权请求返回 403 Forbidden + 明确错误信息

### 4.3 操作日志记录操作人

- 当前日志中操作人统一写 admin，需改为实际登录用户名
- 涉及所有写操作的日志记录点
- 日志表如果没有 operator 字段，需要添加

## 五、数据库变更

### users 表扩展
- 新增字段：`display_name` VARCHAR(64)
- 新增字段：`role` VARCHAR(20) DEFAULT 'operator'（admin/operator/viewer）
- 新增字段：`status` VARCHAR(20) DEFAULT 'active'（active/disabled）
- 新增字段：`last_login_at` DATETIME
- 现有 admin 用户自动设为 role='admin'

### 操作日志表
- 确认是否有 operator/username 字段
- 如无，新增 `operator` VARCHAR(64) 字段

6. ✅ 禁用用户无法登录

### 个人设置
7. ✅ 用户可修改自己的密码（需验旧密码）
8. ✅ 用户可修改自己的显示名
9. ✅ 导航栏显示当前用户名和角色
