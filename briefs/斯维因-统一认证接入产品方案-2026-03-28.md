## 三、对接接口（认证平台已提供）

### 1. 发起微信登录
```http
GET https://auth.aiusing.net/api/auth/wechat/redirect?redirect_uri=<工作台回调地址>
```
用途：工作台将用户重定向到统一认证平台发起微信授权。

### 2. 微信回调
```http
GET https://auth.aiusing.net/api/auth/wechat/callback?code=xxx&state=xxx
```
用途：平台自动处理，工作台无需直接调用；平台处理后会将 token 带在 `redirect_uri` 的 fragment 中回跳工作台。

### 3. 在线验证 token
```http
GET https://auth.aiusing.net/api/auth/verify
Authorization: Bearer <JWT_TOKEN>
```
返回示例：
```json
{
  "valid": true,
  "account_id": "xxx",
  "expires_at": "2026-03-28T10:00:00Z"
}
```

### 4. 查询插件授权
```http
GET https://auth.aiusing.net/api/license/check?plugin=<plugin_key>
Authorization: Bearer <JWT_TOKEN>
```
返回示例：
```json
{
  "product": "data-ops-workbench",
  "plugin": "plugin_ai_assistant",
  "authorized": true,
  "expires_at": "2026-12-31T00:00:00Z"
}
```

### 5. 下载公钥
```http
GET https://auth.aiusing.net/public/keys/jwt_public.pem
```
用途：工作台下载公钥后，可在本地使用 RS256 验证 JWT。

## 四、用户流程设计

### 场景 A：未登录用户进入工作台
1. 用户打开工作台
2. 工作台检测本地无有效 token
3. 展示统一认证登录入口
4. 用户点击“微信登录”
5. 浏览器跳转统一认证平台微信授权地址
6. 授权完成后回跳工作台回调页
7. 回调页解析 fragment 中的 token
8. 工作台调用 `/api/auth/verify` 校验 token
9. 校验通过后建立本地登录态并进入首页

### 场景 B：已登录用户再次打开工作台
1. 工作台读取本地 token
2. 优先进行在线验证
3. 在线不可用时，允许尝试本地离线验签
4. 验签通过则保留受控登录态；如策略不允许则提示重新登录

### 场景 C：进入受控插件功能
1. 用户进入插件功能页面
2. 工作台调用 `/api/license/check?plugin=<plugin_key>`
3. 若已授权，则正常放行
4. 若未授权，则展示未授权提示页/提示区

## 五、界面与交互要求

### 1. 登录页
最少包含：
- 产品名称 / Logo
- “微信登录”按钮
- 说明文案：通过统一认证平台完成身份校验
- 错误提示区
- 服务异常提示区

### 2. 回调处理中间页
需要具备：
- 正在登录中提示
- token 解析
- verify 调用
- 成功跳首页
- 失败回登录页

### 3. 用户状态区
建议放置于右上角或设置页：
- 当前登录状态
- `account_id`
- token 到期时间
- 认证来源：统一认证平台
- 退出登录

### 4. 授权提示
对于受控插件功能：
- 已授权：正常进入
- 未授权：展示“当前账号未开通此功能”
- 如接口返回到期时间，则显示到期时间

## 八、分阶段验收标准

### 阶段 1：登录闭环
验收标准：
- 用户可完成一次完整微信登录
- 工作台可正确获得 token
- `/api/auth/verify` 校验通过
- 刷新后登录态可恢复

### 阶段 2：授权闭环
验收标准：
- 至少 1 个插件接入授权校验
- 已授权与未授权分流正确
- 页面提示清晰

### 阶段 3：离线验签
验收标准：
- 工作台可下载公钥
- 工作台可使用本地公钥校验 JWT 签名
- 认证平台短时不可用时，有明确降级策略

### 阶段 4：双平台打包验证
验收标准：
- Linux 包可启动、可登录、可访问至少 1 个授权功能
- Windows 包可启动、可登录、可访问至少 1 个授权功能
