# v3.4.2 — Windows 版本问题修复

## 问题 1：启动器 exe 无反应
- 用户双击「数据运维工作台.exe」没有任何反应，管理员权限也不行
- 可能原因：
  - Nuitka --onefile 模式下 pystray/tkinter 依赖没正确打入
  - 启动器需要的 DLL 缺失
  - --windows-disable-console 导致错误信息不可见
- 修复方向：
  - CI 中用 --standalone 模式替代 --onefile，确保依赖完整
  - 或者换方案：launcher 不用 Nuitka 编译，改用 PyInstaller 打包（更成熟的 GUI 打包方案）
  - 或者更简单：launcher 用纯 bat 包装成 vbs 脚本隐藏窗口，不依赖额外编译
  - 最优方案：launcher.py 直接随主服务一起编译进 standalone 目录，用同一套依赖

## 问题 2：服务启动慢 + 拒绝访问
- start.bat 启动后能看到 localhost:8580 但服务还没完全就绪
- 用户立即访问会被拒绝，等一会儿又能访问
- 关闭终端再打开又拒绝 — 说明上一个进程没完全退出或端口被占用
- 修复方向：
  - 启动后轮询健康检查接口 /api/health，就绪后再提示用户
  - 启动时检测端口占用，如果 8580 已被占用提示用户或自动 kill 旧进程
  - 显示明确的「服务启动中，请稍候...」提示

## 问题 3：启动时自动拉起浏览器
- 用户期望：程序启动后自动打开浏览器访问 http://localhost:8580
- start.bat 中服务就绪后自动 `start http://localhost:8580`
- 实现：启动 dataops-server 后台进程后，循环检测 /api/health，通后自动打开浏览器

## 修复后的 start.bat 期望行为
1. 启动 dataops-server（后台）
2. 显示「服务启动中，请稍候...」
3. 轮询 /api/health 直到返回 200
4. 显示「服务已就绪」
5. 自动打开浏览器 http://localhost:8580
6. 保持窗口运行（显示日志），Ctrl+C 停止

## 验收标准
1. ✅ start.bat 启动后自动打开浏览器
2. ✅ 浏览器打开时服务已就绪，不会拒绝访问
3. ✅ 启动器 exe 能正常运行（或暂时降级为可靠方案）
4. ✅ 关闭终端后再启动不会端口冲突
